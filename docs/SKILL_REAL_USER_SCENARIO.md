# Skill 실제 사용자 시나리오 — 명세 + 실행 체크 원장

작성일: `2026-06-14 KST` · 대상: `vibeloop-harness` Skill

이 문서는 **"진짜 사용자가 스킬을 쓸 때의 완전한 시나리오"** 를 명세하고, **스킬이 실제로 돌 때마다 단계별로 체크**할 수 있게 만든 살아있는 문서다. 두 가지를 동시에 만족시킨다.

1. **명세**: 목표 28단계 흐름 + 각 단계의 환경 전제 + 기대 결과 + 검증 방법(report 필드).
2. **진화**: 단계가 실제 구현·실측되면 상태(❌→⚠️→✅)를 올리고, 실행할 때마다 **Run Ledger**에 결과를 1행씩 누적한다.

> 과장 금지 규칙: 모든 단계 상태는 **코드 검증 근거**로만 적는다. fixture/scripted 결과를 "실사용자 PASS"로 부르지 않는다. (참조: [implement_20260614_193356.md](../dev-plan/implement_20260614_193356.md), [CODEX_SKILL_FULL_UAT_RUNBOOK.md](./CODEX_SKILL_FULL_UAT_RUNBOOK.md))

상태 범례: ✅ 구현·실측 / ⚠️ 부분(라이브러리/래퍼만) / ❌ 목표(미구현).

## 0. 실사용자 run 합격 공식

```text
REAL_USER_RUN_PASS =
  real_llm_builder_used            # mock/script 아님(실 codex/LLM)
  ∧ orchestrator_read_skill_md     # LLM이 SKILL.md 읽고 모드 분기·task/eval 생성
  ∧ one_issue_selected_with_reason # 한 번에 1개 + 선택 근거 기록
  ∧ deterministic_gates_only       # accept/select는 고정 게이트+Arbiter (LLM 아님)
  ∧ pr_candidate ⇔ (selected ∧ accept ∧ ALL_PASS ∧ qualified)
  ∧ no_auto_merge                  # draft PR 후보까지만
  ∧ false_pass == 0 ∧ leak == 0
```

현재 이 공식 중 **`real_llm_builder_used` + `deterministic_gates_only` + `pr_candidate ⇔ …` + `no_auto_merge` + `false_pass==0 ∧ leak==0`은 실 GitHub repo + draft PR로 실측됨**(Run Ledger R2/R3/R11/R13). `orchestrate`는 발견→선택→task 자동생성→issue별 `runImprovementLoop`까지 결정론 코어 명령으로 검증됐고, 단일 이슈 local branch는 `improve --promote-branch`, 단일 이슈 GitHub draft PR은 `improve --github-draft-pr`로 가능하다. R7/R8은 실제 Codex Skill orchestrator가 `SKILL.md`를 읽고 prompt runner를 호출했음을 fixture builder로 증명했고, R9/R10은 같은 Skill prompt 흐름에 **실 gpt-5.5 builder**를 결합해 user_issue와 auto_discovery 모두 local promotion/reverify까지 통과했다. R11은 실 Codex `orchestrate`가 실 GitHub repo에서 2개 이슈를 자동 발견→수정→검증→stacked draft PR까지 생성해 **live RU-3 verification PASS**를 남겼다. R13은 verbose comparator + 실 Codex challenger 구도로 2개 자동발견 이슈 모두 `strict_fixed_score_win`과 `real_codex_challenger_selected_every_issue=true`를 만족해 **controlled RU-3 full improvement PASS**를 남겼다. 다만 hidden/adversary semantic gate와 범용 프로젝트 corpus는 아직 PASS가 아니다.

| 수정 후 재탐색       | `--promote-branch`에서 issue 후 다시 discovery                                                                                                           | 각 issue 후 다시 discovery 실행                                                                                             | 충족(로컬)               |
| 코어 PR브랜치        | `improve` local+GitHub draft PR 가능, `orchestrate --promote-branch --github-draft-pr` stacked PR 가능                                                   | 실 Codex/GitHub evidence                                                                                                    | 충족(R11 live verification) |
| live Codex full run  | R11에서 실 Codex + GitHub verification, R13에서 controlled RU-3 strict-best full improvement evidence 확보                                                  | adversary semantic gate + hidden/adversary eval까지 포함한 product-level evidence                                         | strict-best controlled 충족 / adversary 미충족 |

**결정**: 현재 `vibeloop orchestrate --promote-branch`는 **local RU-3 substrate**로 격상한다. R11로 `apply selected patch → commit/integration branch → rediscover → next issue → GitHub draft PR 후보`가 한 `orchestrate` 실 run evidence로 남았다. R13으로 controlled RU-3 2이슈 strict fixed-score full improvement도 확인됐다. 단 product-level full autonomy/실사용자 100% 승격은 adversary semantic gate와 hidden/adversary eval 고도화 후에만 한다.

`VIBELOOP_UAT_ALLOW_VERIFICATION_ONLY=1`은 RU-2에서 verification pass 상태를 관찰하기 위한 옵션일 뿐, 실패 run을 성공으로 바꾸지 않는다. `verification_status=fail`이면 항상 exit non-zero다.
