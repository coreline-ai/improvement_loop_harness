# Skill 실제 사용자 시나리오 — 명세 + 실행 체크 원장

작성일: `2026-06-14 KST` · 대상: `vibeloop-harness` Skill

이 문서는 **"진짜 사용자가 스킬을 쓸 때의 완전한 시나리오"** 를 명세하고, **스킬이 실제로 돌 때마다 단계별로 체크**할 수 있게 만든 살아있는 문서다. 두 가지를 동시에 만족시킨다.

1. **명세**: 목표 28단계 흐름 + 각 단계의 환경 전제 + 기대 결과 + 검증 방법(report 필드).
2. **진화**: 단계가 실제 구현·실측되면 상태(❌→⚠️→✅)를 올리고, 실행할 때마다 **Run Ledger**에 결과를 1행씩 누적한다.

> 과장 금지 규칙: 모든 단계 상태는 **코드 검증 근거**로만 적는다. fixture/scripted 결과를 "실사용자 PASS"로 부르지 않는다. (참조: [implement_20260614_193356.md](../dev-plan/implement_20260614_193356.md), [CODEX_SKILL_FULL_UAT_RUNBOOK.md](./CODEX_SKILL_FULL_UAT_RUNBOOK.md))

상태 범례: ✅ 구현·실측 / ⚠️ 부분(라이브러리/래퍼만) / ❌ 목표(미구현).

## 0-0. 현재 목적 잠금 — 내부 루프 우선

2026-06-23 이후 현재 작업의 목적은 CI/corpus/semantic domain 확장이 아니다. 이 프로젝트의 목적은 **내부 개선 루프가 실제 코드 문제를 하나씩 해결하며 더 나은 결과를 선택하는 것**이다. GitHub draft PR은 선택된 개선분을 출판하는 도구이고, CI는 회귀 확인 도구일 뿐이다.

현재 실행 순서는 아래로 고정한다.

1. 실제 내부 코드 개선 issue 1개 확정
2. 루프가 후보 생성/검증/선택까지 도는지 확인
3. 최종 선택 개선분만 branch로 분리
4. GitHub draft PR 생성
5. PR diff와 evidence로 “이게 최종 개선분”임을 확인

관련 계획 문서: [implement_20260623_082118.md](../dev-plan/implement_20260623_082118.md)

현재 중단 범위: 새 CI artifact 재현성 기능, broad corpus 추가 확장, semantic/M4 corpus 신규 domain 확장. 이 항목들은 사용자가 다시 명시하기 전까지 현재 작업 queue가 아니다.

## 0. 실사용자 run 합격 공식

```text
REAL_USER_RUN_PASS =
  real_llm_builder_used            # mock/script 아님(실 codex/LLM)
  ∧ orchestrator_read_skill_md     # LLM이 SKILL.md 읽고 모드 분기·task/eval 생성
  ∧ one_issue_selected_with_reason # 한 번에 1개 + 선택 근거 기록
  ∧ deterministic_gates_only       # accept/select는 고정 게이트+Arbiter (LLM 아님)
  ∧ pr_candidate ⇔ (selected ∧ accept ∧ ALL_PASS ∧ qualified)
  ∧ no_auto_merge                  # draft PR 후보까지만
  ∧ false_pass == 0 ∧ leak == 0
```

현재 이 공식 중 **`real_llm_builder_used` + `deterministic_gates_only` + `pr_candidate ⇔ …` + `no_auto_merge` + `false_pass==0 ∧ leak==0`은 실 GitHub repo + draft PR로 실측됨**(Run Ledger R2/R3/R11/R13/R14/R15/R16/R17/R28). `orchestrate`는 발견→선택→task 자동생성→issue별 `runImprovementLoop`까지 결정론 코어 명령으로 검증됐고, 단일 이슈 local branch는 `improve --promote-branch`, 단일 이슈 GitHub draft PR은 `improve --github-draft-pr`로 가능하다. R7/R8은 실제 Codex Skill orchestrator가 `SKILL.md`를 읽고 prompt runner를 호출했음을 fixture builder로 증명했고, R9/R10은 같은 Skill prompt 흐름에 **실 gpt-5.5 builder**를 결합해 user_issue와 auto_discovery 모두 local promotion/reverify까지 통과했다. R11은 실 Codex `orchestrate`가 실 GitHub repo에서 2개 이슈를 자동 발견→수정→검증→stacked draft PR까지 생성해 **live RU-3 verification PASS**를 남겼다. R13은 verbose comparator + 실 Codex challenger 구도로 2개 자동발견 이슈 모두 `strict_fixed_score_win`과 `real_codex_challenger_selected_every_issue=true`를 만족해 **controlled RU-3 full improvement PASS**를 남겼다. R14는 P0 복구 후 단일 이슈 live rerun과 durable evidence bundle 보존을 확인했다. R15/R16/R17은 Python stdlib, JS monorepo scope, React-like/Django-like/Rails-like/Android-like 대표 셀에서 실 Codex/GitHub draft PR과 hidden acceptance를 확인했다. R28은 실제 repo 기존 source repair temp clone 결과를 GitHub private evidence repo + draft PR까지 publish/검증했다. R32는 실제 repo temp clone의 invoice-total business fixture repair를 실 Codex + hidden verifier로 검증했고, R33은 그 business fixture repair를 credentialed CI artifact + downloaded audit으로 재현했다. > R13 해석: **controlled RU-3 strict-best full improvement lane이 실측 통과**했다. 자동 발견 2이슈 모두 verbose comparator와 실 Codex challenger가 후보로 들어갔고, 두 이슈 모두 실 Codex challenger가 `strict_fixed_score_win`으로 선택됐다. 이 증거는 “multi-issue 최고수정 선택” 미완을 controlled fixture 범위에서 닫는다. 이후 controlled adversary semantic live evidence와 local real reviewer generation evidence도 확보됐지만, broad semantic/M4 corpus와 범용 프로젝트 corpus는 아직 후속이다.
>
> R14 해석: **P0 실행환경 복구 후 RU-1 live rerun과 evidence bundle 보존이 실측 통과**했다. `corepack pnpm uat:live-preflight`가 필수 항목을 통과했고, 실제 Codex/GitHub run이 draft PR을 만든 뒤 `~/.vibeloop/uat-evidence/...`에 ledger/report/proxy/run manifest를 보존했다. 이 행은 증거 보존 검증이며, RU-3/full/adversary semantic PASS로 격상하지 않는다.
>
> R15 해석: **Node 외 toolchain 대표 셀(Python stdlib)이 실 Codex/GitHub draft PR까지 통과**했다. 첫 시도에서는 visible regression만 만족하고 default quantity hidden gate에서 fail-closed되어 PR을 만들지 않았고, task 의미를 명확히 한 재실행에서 `item.get("quantity", 1)` 수정 + regression test + hidden acceptance + final reverify + draft PR 검증이 통과했다. 이 행은 P5 대표 live 증거이며, monorepo/중간규모/실제 대형 서비스 corpus나 제품 전체 100% PASS가 아니다.
>
> R16 해석: **모노레포 scope boundary 대표 셀이 실 Codex/GitHub draft PR까지 통과**했다. `packages/cart/`만 write scope로 열고 `packages/catalog/`를 forbidden/protected로 둔 상태에서 실제 Codex 후보가 cart 소스와 cart regression test만 바꿨고, protected/diff scope·hidden acceptance·final reverify·draft PR 파일 검증이 모두 통과했다. 이 행은 P5 대표 구조 live 증거이며, 실제 대형 monorepo나 범용 서비스 corpus PASS로 격상하지 않는다.
>
> R17 해석: **React-like/Django-like/Rails-like/Android-like framework-shaped 대표 셀 4개가 실 Codex/GitHub draft PR까지 통과**했다. 네 repo 모두 private GitHub repo에서 buggy base를 만들고 실 gpt-5.5 후보 2개를 생성했으며, selected patch가 hidden acceptance·final reverify·draft PR 파일 검증·main unchanged를 통과했다. React-like 셀 추가 과정에서 후보마다 기존 base test 수정 여부가 달라지는 자연 변이가 드러나 verifier를 필수 파일 + optional 허용 파일 계약으로 보강했고, 재실행에서 `BROAD_LIVE_REPRESENTATIVE_PASS`가 확인됐다. 이 행은 P5 representative broad framework evidence이며, 실제 대형 서비스나 임의 사용자 repo 전체 PASS로 격상하지 않는다.
>
> R5/R6 정리: 둘은 중복 PASS가 아니라 RU-2 verification lane의 별도 실행 기록이다. R5는 archive 정리 run, R6은 remote 보존(`KEEP_REMOTE=1`) rerun이며 둘 다 scripted issue queue라 RU-3 자동발견/full improvement 증거로 격상하지 않는다.

### 기록 행 작성 규칙

- 실행마다 위 표에 1행 추가. 모드(지정/자동), 빌더가 mock인지 실LLM인지 **반드시 명시**.
- "통과 단계"는 §2의 # 중 실제 충족된 것만. 미충족은 적지 않는다.
- 새 live run의 증거는 stdout ledger의 `evidence.evidence_bundle`/`evidence_manifest` 보존 경로와 실제 `selection-report.json`/`eval-report.json`/`quality-report.json`/reverify/proxy stats 경로를 함께 적는다. 과거 `KEEP_TMP=1` 로컬 경로만 남은 행은 "과거 방식"으로 유지한다.
- false pass(부당 accept)·leak이 0이 아니면 그 run은 FAIL로 기록.

**결정**: 현재 `vibeloop orchestrate --promote-branch`는 **local RU-3 substrate**로 격상한다. R11로 `apply selected patch → commit/integration branch → rediscover → next issue → GitHub draft PR 후보`가 한 `orchestrate` 실 run evidence로 남았다. `VIBELOOP_UAT_ALLOW_VERIFICATION_ONLY=1`은 RU-2에서 verification pass 상태를 관찰하기 위한 옵션일 뿐, 실패 run을 성공으로 바꾸지 않는다. `verification_status=fail`이면 항상 exit non-zero다.

기본 동작:

- P2 Postgres evidence는 DB URL/연결 확인만으로 통과하지 않는다. `prisma_store_smoke`가 실제 PrismaStore candidate round-trip, 보안 메타데이터 round-trip, duplicate fingerprint 거부를 모두 `pass`로 남겨야 release gate와 CI artifact audit이 통과한다.
- 각 live UAT는 stdout ledger의 `evidence.evidence_bundle`에 내구 보존 경로를 출력한다. 기본 위치는 `~/.vibeloop/uat-evidence/`이고, CI나 감사 run에서는 `VIBELOOP_UAT_EVIDENCE_DIR=$PWD/.uat-evidence`처럼 바꿀 수 있다.
- CI는 `workflow_dispatch` 수동 실행을 지원한다. CI `prisma-store` job은 실제 PostgreSQL service에서 `pnpm uat:postgres-contract`를 실행하고 `.uat-evidence-postgres`를 `postgres-contract-evidence-*` artifact로 업로드한다. CI `adversary-live` job은 `node:22-alpine`을 preload한 Docker runner에서 `pnpm uat:adversary-live`를 실행하고 `.uat-evidence-adversary`를 `adversary-live-evidence-*` artifact로 업로드한다. CI `release-evidence-audit` job은 `postgres-contract-evidence-*`, `adversary-live-evidence-*`, `uat-evidence-*`를 다운로드/병합한 뒤 `release-evidence-audit`으로 기본 P2/P4/P5 ledger status, manifest, Postgres checks, adversary attack/safety/reviewer provenance ledger, repo matrix cell summary를 preflight 없이 검사하고, Step Summary와 `release-evidence-audit-*` artifact에 감사 JSON/stderr를 남긴다. evidence artifact가 없거나 다운로드 단계가 실패해도 audit step은 계속 실행되어 missing evidence JSON을 남긴다. 수동 감사에서는 `uat:release-evidence-audit:gh -- --latest`가 최근 CI run들을 훑어 evidence artifact가 있는 첫 run을 내려받아 같은 감사기를 실행하고, `--repo`나 `GITHUB_REPOSITORY`가 있으면 각 후보 run의 artifact 목록을 먼저 조회해 `missing_artifacts`, `replay_ready_matching_count`, artifact `digest`/`size_in_bytes`/`expired`를 남긴다. 이름만 맞지만 만료·빈 파일·sha256 digest 누락으로 재현 불가능한 artifact는 다운로드 전에 건너뛰며, 특정 run만 고정하려면 `--run-id <run-id>`를 쓴다. `--all-release-evidence`로 P3 live Codex와 P5 representative live evidence(Python, monorepo, broad framework)까지 포함하거나 `--scenario <scenario>`로 특정 evidence만 감사할 수 있다.
- `tmp_root`도 기본 보존한다. 디스크 정리가 필요할 때만 `VIBELOOP_UAT_PRUNE=1`을 붙인다. 기존 `VIBELOOP_UAT_KEEP_TMP=1`은 하위 호환 override로 남아 있으며, 둘 다 있으면 prune하지 않는다.
- 보존 번들에는 `uat-evidence-manifest.json`, `ledger.json`, `reports/`, `runs/`, `logs/`, `proxy/proxy-stats.json`이 포함된다. 복사된 run manifest는 `audit_keep: true`로 표시된다.
