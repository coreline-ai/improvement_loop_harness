## 개발 목적

VibeLoop Harness의 검증 커널을 CLI로 먼저 완성하고(MVP-0), 그 위에 single agent loop(MVP-1), 웹 report viewer(MVP-2), GitHub Draft PR(MVP-3)을 순서대로 올린다. 커널의 완성 기준은 "하나의 candidate patch를 `accept | reject | needs_human_review | needs_more_tests`로 deterministic하게 판정하고 eval-report.json을 항상 남기는 것"이다.

## 개발 범위

- Phase 1~11: MVP-0 CLI 검증 커널 (모노리포 → task-protocol → artifacts → workspace-runner → guards → eval 코어 → 증거 검증 → decision/report → agent adapter/proxy → CLI 배선 → fixture e2e)
- Phase 12: MVP-1 마감 (정적 HTML report, retention gc)
- Phase 13~14: MVP-2 (Prisma DB + Fastify API, Next.js report viewer)
- Phase 15: MVP-3 (GitHub Draft PR)
- Phase 16~17: MVP-4 자율 개선 루프 — 봇이 문제를 발견하고, 1개씩 순차로 수정 루프를 돌리고, 고정 결과지 통과분만 PR로 만들고, 반복 ([AUTONOMOUS_LOOP_SPEC.md](../docs/AUTONOMOUS_LOOP_SPEC.md))

## 공통 진행 규칙

- 각 Phase는 앞선 Phase의 자체 테스트 완료 후에만 시작한다.
- 구현 중 발생한 이슈는 해당 Phase에서 수정하고 기록한다.
- 체크박스 상태를 실제 진행 상태와 맞게 업데이트한다.
- 문서에 없는 범위 확장은 하지 않는다.
- Phase 완료마다 단일 커밋(`feat(phase-N): ...`)을 만든다.

## 문서 기반 제약 사항

- 웹 UI/DB CRUD부터 구현 금지 — Phase 13~14는 Phase 11(e2e) 통과 후에만.
- 가드 통과 전 프로젝트 명령 게이트 실행 금지 (fail-fast, EVAL_ENGINE_SPEC §3).
- 가드는 하네스 내장(builtin) — 대상 repo 스크립트 위임 금지.

- worktree/artifact를 대상 repo 내부에 배치 금지 — `~/.vibeloop` 데이터 디렉터리.
- agent에게 eval-report.json 직접 작성시키기 금지.
- `docs/`·`schemas/`가 source of truth — 코드와 충돌 시 스펙 개정 절차 선행.
- 기술 스택 고정: TypeScript + Node 22, pnpm workspaces, vitest, execa, ajv(2020-12), Fastify, Prisma+PostgreSQL(Phase 13부터), git ≥ 2.31.
