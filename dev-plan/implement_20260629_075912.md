# implement_20260629_075912.md — Local Gitea fast lane + P1 test speedup

작성 일시: `2026-06-29 07:59:12 KST`

이 문서는 P1 자연어 Skill UX 검증을 더 빠르게 반복하기 위한 개발 계획이다. 핵심은 외부 GitHub 반복 호출을 개발 루프에서 분리하고, 로컬 Gitea 기반 PR-like fast lane과 targeted corpus 실행 경로를 추가해 "수정 직후 빠른 확인"과 "최종 GitHub 증거"를 명확히 나누는 것이다.

리뷰 반영 상태: `Needs revision` 판정이 나온 항목을 이 문서에 반영한다. 구현 착수 전 반드시 provider/evidence/audit 계약을 먼저 고정한다.

## 개발 목적

- P1 검증 속도를 높인다.
- 로컬 Gitea를 설치/기동해 Git branch push + pull request 생성/조회 흐름을 로컬에서 빠르게 검증한다.
- 기존 GitHub draft PR 최종 증거는 유지하되, 매 수정마다 GitHub 56-variant full lane을 반복하지 않도록 fast/targeted/full/final lane을 분리한다.
- "루프 동작 검증"과 "제품 전체 100% PASS 증거"를 섞지 않는다.

## 개발 범위

- 로컬 Gitea 개발 인프라:
  - `infra/gitea/docker-compose.yml`
  - Gitea bootstrap script 또는 UAT helper
  - local user/token/repo 생성 자동화
- Git provider 추상화:
  - 기존 `@vibeloop/github-integration`의 GitHub draft PR 계약은 보존
  - Gitea PR-like provider 추가
  - provider별 PR 생성/조회 응답은 공통 내부 조작 결과로 정규화하되, evidence claim은 provider별로 분리
  - provider result는 discriminated union으로 고정
    - `provider: "github"`: GitHub draft PR strict evidence만 표현
    - `provider: "gitea"`: local PR-like evidence만 표현
- P1 fast scripts:
  - representative 2-variant smoke
  - targeted variant 실행
  - full local 56-variant 실행
  - local Gitea PR-like corpus 실행
  - 최종 GitHub draft PR smoke 실행
- Evidence/audit:
  - Gitea evidence는 `local_pr_like=true`, `github_draft_pr=false`, `github_draft_pr_verified=false`로 분리
  - GitHub evidence는 기존 `github_draft_pr_verified=true` 계약 유지
  - release/documentation에서 둘을 절대 동일 PASS로 합치지 않음
  - `gitea_pr_like`는 canonical field로 사용하지 않음
- 문서:
  - README P1 명령 정리
  - RELEASE_GATE_MATRIX에 Gitea fast lane과 GitHub final lane 구분 반영
  - SKILL_REAL_USER_SCENARIO에 증거 범위 truth note 추가

## 제외 범위

- Gitea를 GitHub 최종 증거로 대체하지 않는다.
- CI 구현/CI workflow 확장은 하지 않는다.
- 임의/대형 사용자 repo 전체 PASS를 선언하지 않는다.
- 기존 `coreline-ai/vibeloop-*` GitHub repo 삭제 자동화는 이 계획 범위가 아니다.
- Gitea Actions/runner는 사용하지 않는다. PR 생성/조회 fast lane만 다룬다.
- 제품 UI/서버 운영 기능 추가가 아니다.

## 공통 진행 규칙

- `main`만 사용한다. 새 작업 브랜치를 만들지 않는다.
- 커밋 author/committer는 `coreline-ai <coreline-ai@users.noreply.github.com>`만 사용한다.
- 각 Phase는 자체 테스트를 포함한다.
- 빠른 확인은 Gitea/local lane으로, 최종 호환성은 GitHub lane으로 분리한다.
- Gitea PASS는 `local_pr_like` PASS이지 `GitHub draft PR` PASS가 아니다.
- `local_pr_like`를 canonical field로 사용한다. `gitea_pr_like` alias를 새 evidence에 남기지 않는다.
- 기존 `--github-draft-pr`는 GitHub draft PR 전용이다. Gitea provider와 함께 쓰면 fail-closed로 거부한다.
- Gitea evidence가 `draft_pr=true`, `github_draft_pr=true`, `github_draft_pr_verified=true` 중 하나라도 만들면 audit 실패로 본다.
- `p1-github-final` 계열은 `scope=smoke|full`, `variant_count`, `required_count`를 ledger에 남긴다.
- 실패 시 전체 56개를 재실행하지 말고 실패 variant만 targeted rerun한다.
- `corepack pnpm`을 기본 명령으로 사용한다.
- 문서에 없는 범위 확장은 하지 않는다.

## Phase 4. Evidence/audit separation for Gitea vs GitHub

### 목표

Gitea fast lane evidence가 GitHub draft PR evidence로 오인되지 않도록 audit 계약을 분리한다.

### 구현 태스크

- [x] ledger 필드 분리
  - `git_provider`
  - `local_pr_like`
  - `github_draft_pr`
  - `github_draft_pr_verified`
  - `draft_supported`
  - `scope`
  - `variant_count`
  - `required_count`
  - `run_id`
  - `evidence_root`
- [x] release audit 옵션 추가
  - `--require-skill-prompt-corpus-local-pr-like`
  - 기존 `--require-skill-prompt-corpus-github-pr`는 GitHub만 허용
- [x] live-state audit provider-aware 처리
  - GitHub: `OPEN`, draft, auto-merge null, base/head/body freshness
  - Gitea: open PR, base/head/body freshness, `draft_supported=false` 명시
- [x] chunk-audit도 provider field를 aggregate에 남김
- [x] fail-closed cases 추가
  - `git_provider=gitea`인데 `draft_pr=true`, `github_draft_pr=true`, `github_draft_pr_verified=true` 중 하나라도 있으면 fail
  - Gitea evidence를 `--require-skill-prompt-corpus-github-pr`에 넣으면 fail
  - GitHub evidence를 `--require-skill-prompt-corpus-local-pr-like`에 넣으면 provider mismatch로 fail 또는 not-counted로 고정
  - parent corpus ledger와 child ledger의 `git_provider`가 다르면 fail
  - Gitea live-state audit에서 PR open/base/head/body freshness가 틀리면 fail
  - `draft_supported=false` evidence가 GitHub draft requirement를 만족하면 fail
  - targeted/2-variant smoke evidence가 full 56-variant audit을 만족하면 fail
  - stale evidence root에서 다른 provider 최신 ledger가 선택되면 fail
  - Docker/Colima 부재로 `blocked(exit 20)`인 Gitea preflight는 PASS evidence로 계산하지 않음

## 이번 계획의 성공 기준

- 개발 중 P1 확인은 `p1-fast` 또는 `p1-targeted`로 시작한다.
- PR publish 관련 회귀는 로컬 Gitea로 먼저 확인한다.
- GitHub는 최종 smoke 또는 release evidence 용도로만 사용한다.
- 어떤 문서도 Gitea PASS를 GitHub draft PR PASS나 제품 전체 100% PASS로 표현하지 않는다.
