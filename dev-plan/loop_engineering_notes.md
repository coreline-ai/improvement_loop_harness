# 루프 엔지니어링 정리

## 1. 핵심 개념

**루프 엔지니어링**은 사람이 매번 직접 프롬프트를 작성하는 방식에서 벗어나,
에이전트가 스스로 **조사 → 실행 → 테스트 → 실패 분석 → 개선 → 검증 → 기억**을 반복하도록 만드는 구조입니다.

즉, 핵심은 단순히 “좋은 프롬프트 작성”이 아니라,
**좋은 결과가 나올 때까지 스스로 개선되는 반복 시스템을 설계하는 것**입니다.

사람의 역할은 프롬프트 작성자가 아니라,
**루프 설계자 / 목표 정의자 / 최종 승인자**에 가까워집니다.

---

## 2. 문제의식

기존 방식은 사람이 직접 프롬프트를 만들고, 결과를 확인하고, 다시 수정하는 구조입니다.

하지만 앞으로는 다음과 같은 방식이 더 중요해집니다.

```text
사람이 직접 프롬프트를 작성하는 것보다
에이전트가 프롬프트를 작성하고,
그 프롬프트를 스스로 테스트하고,
실패하면 개선하고,
검증 결과를 기억하게 만드는 구조
```

핵심 변화는 다음과 같습니다.

```text
기존 방식:
사람 → 프롬프트 작성 → AI 실행 → 사람이 결과 확인 → 사람이 수정

루프 엔지니어링 방식:
사람 → 목표 정의 → 에이전트가 프롬프트 생성 → 실행 → 테스트 → 실패 분석 → 개선 → 검증 → 메모리 저장 → 반복
```

---

## 3. 루프의 기본 구조

```text
1. 목표 정의
   ↓
2. 에이전트가 조사
   ↓
3. 해결 방법 또는 프롬프트 생성
   ↓
4. 테스트 케이스 생성
   ↓
5. 실제 테스트 수행
   ↓
6. 문제점 기록
   ↓
7. 실패 원인 분석
   ↓
8. 개선 프롬프트 또는 수정안 작성
   ↓
9. 개별 수정 및 재테스트
   ↓
10. 전체 테스트 재실행
   ↓
11. 성공/실패 패턴을 메모리에 저장
   ↓
12. 종료 조건 충족 여부 확인
   ↓
13. 부족하면 반복
```

이 구조에서 중요한 것은 “반복” 그 자체가 아니라,
**반복할 때마다 남는 자산이 있어야 한다는 점**입니다.

---

## 4. 루프 엔지니어링의 핵심 질문

### 4.1 신뢰할 만한 평가 도구가 있는가?

평가를 하나의 AI에게만 맡기면 위험합니다.
작성한 에이전트가 자기 결과를 직접 평가하면 자기합리화, 누락, 과신이 발생할 수 있습니다.

따라서 **검증자와 평가자 분리**가 필요합니다.

```text
작성자 Agent
- 해결안 작성
- 코드 수정
- 프롬프트 생성
- 실행 계획 수립

검증자 Agent
- 결과가 실제로 동작하는지 확인
- 테스트 케이스 실행
- 오류 재현
- 증거 기반으로 성공/실패 판단

평가자 Agent
- 목표 기준에 맞는지 판단
- 품질 점수화
- 부족한 부분 정리
- 개선 우선순위 제안

사람
- 신규 기능 승인
- 방향성 통제
- 종료 여부 최종 판단
- 요구사항 충돌 시 의사결정
```

핵심 원칙은 다음과 같습니다.

```text
작성자와 검증자는 분리한다.
검증자와 평가자도 가능하면 분리한다.
자동 평가 결과가 애매하면 사람이 판단한다.
```

---

### 4.2 루프 한 바퀴를 돌고 남는 것이 있는가?

좋은 루프는 한 번 돌고 끝나는 것이 아니라, 반드시 자산이 남아야 합니다.

루프 한 바퀴 후 남아야 하는 것은 다음과 같습니다.

```text
- 실행 결과
- 테스트 결과
- 실패 목록
- 실패 원인
- 수정 내역
- 개선된 프롬프트
- 통과한 테스트 케이스
- 새로 발견된 테스트 케이스
- 다음에 참고할 메모리
- 반복하지 말아야 할 실수
- 신규 기능 제안 후보
```

즉, 루프가 돌 때마다 단순히 결과만 좋아지는 것이 아니라
**테스트 자산, 실패 기록, 개선 규칙, 메모리**가 함께 쌓여야 합니다.

---

### 4.3 조사 → 검증 → 정립 → 참조 구조가 있는가?

루프는 단순 실행 반복이 아니라 다음 구조를 가져야 합니다.

```text
조사
- 문제 이해
- 기존 사례 확인
- 가능한 해결 방법 탐색
- 요구사항과 제약 조건 정리

검증
- 실제 테스트
- 결과 비교
- 실패 여부 확인
- 재현 가능성 확인

정립
- 성공한 방식 정리
- 실패한 방식 제외
- 재사용 가능한 규칙화
- 개선된 프롬프트 저장

참조
- 다음 루프에서 기존 실패 기록 참조
- 같은 실수 방지
- 이전 성공 패턴 재사용
- 개선된 기준으로 재실행
```

이 구조가 있어야 루프가 단순 반복이 아니라 **학습 루프**가 됩니다.

## 5. 자유개선 루프 아이디어

사용자가 정리한 자유개선 아이디어는 다음과 같이 구조화할 수 있습니다.

```text
1. 테스트 케이스 생성
   - 현재 기능을 기준으로 기본 테스트 케이스 생성
   - AI에게 엣지 케이스, 실패 케이스, 사용자 실수 케이스까지 확장하게 함

2. 전체 테스트 수행
   - 모든 테스트 케이스를 AI 또는 자동화 도구가 실제로 확인
   - 단순 추측이 아니라 실행 결과 기반으로 판단

3. 문제점 기록
   - 실패한 테스트
   - 재현 조건
   - 예상 결과
   - 실제 결과
   - 원인 추정
   - 심각도 기록

4. 문제 하나씩 수정
   - 여러 문제를 한꺼번에 고치지 않음
   - 하나 수정 → 해당 테스트 재실행
   - 통과하면 다음 문제로 이동

5. 전체 회귀 테스트
   - 모든 수정이 끝난 후 전체 테스트 재실행
   - 기존 기능이 깨졌는지 확인

6. 테스트 케이스 재확장
   - 수정된 결과를 기준으로 AI가 새로운 테스트 케이스 추가
   - 숨은 문제, 경계 조건, 사용성 문제 탐색

7. 신규 문제 기록
   - 확장 테스트에서 나온 문제를 다시 기록
   - 반복 수정 대상으로 등록

8. 신규 기능 제안
   - 문제 기록 과정에서 단순한 개선 기능을 제안
   - 단, 기능 추가는 자동 진행하지 않음

9. 휴먼 승인
   - 신규 기능은 반드시 사람이 확인 후 진행
   - 루프가 마음대로 개발 범위를 늘리지 못하게 제한
```

---

## 6. 신규 기능 제안 규칙

루프가 문제를 발견하다 보면 자연스럽게 기능 개선 아이디어가 나올 수 있습니다.
하지만 이것을 자동으로 구현하게 하면 개발 범위가 계속 커지는 문제가 생깁니다.

따라서 규칙이 필요합니다.

```text
자동 진행 가능:
- 버그 수정
- 테스트 보강
- 문서화 개선
- 실패 기록 정리
- 프롬프트 개선
- 단순 리팩토링 후보 제안

자동 진행 금지:
- 신규 기능 구현
- 기존 UX 방향 변경
- 데이터 구조 변경
- 외부 API 또는 비용 발생 구조 추가
- 보안 정책 변경
- 사용자의 승인 없는 범위 확장
```

가장 중요한 기준은 다음입니다.

```text
신규 기능 제안은 가능하지만,
신규 기능 구현은 반드시 휴먼 확인 후에만 진행한다.
```

이 규칙이 있어야 루프가 통제 가능한 시스템이 됩니다.

## 7. 명확한 종료 조건

루프는 무한 반복되면 안 됩니다.
반드시 종료 조건이 있어야 합니다.

### 7.1 필수 종료 조건

```text
- 모든 핵심 테스트 케이스 통과
- 치명적 오류 0개
- 주요 기능 회귀 오류 0개
- 문서화된 요구사항 충족
- 평가자 Agent가 기준 이상 점수 부여
- 검증자 Agent가 재현 가능한 실패 없음 확인
```

### 7.2 반복 제한 조건

```text
- 최대 반복 횟수 도달
- 더 이상 새로운 실패가 나오지 않음
- 개선 대비 비용이 커짐
- 동일 유형 실패가 반복되어 사람 판단 필요
- 테스트 기준 자체가 불명확해짐
```

### 7.3 사람 개입 조건

```text
- 신규 기능 제안 발생
- 요구사항 충돌 발생
- 테스트 기준이 불명확함
- 자동 판단 신뢰도가 낮음
- 보안/비용/아키텍처 변경이 필요함
```

### 7.4 현실적인 종료 기준 예시

```text
1. 핵심 테스트 100% 통과
2. 확장 테스트 주요 항목 통과
3. 치명적/중요 버그 없음
4. 신규 기능은 백로그로만 기록
5. 최종 검증자와 평가자의 판단이 일치
6. 사람이 종료 승인
```

## 8. 루프 엔지니어링의 구성 요소

```text
Goal Spec
- 에이전트에게 전달할 명확한 목표

Task Planner
- 작업을 단계별로 나누는 역할

Prompt Generator
- 목표에 맞는 실행 프롬프트 생성

Executor
- 실제 작업 수행

Test Case Generator
- 기본/확장/엣지 테스트 케이스 생성

Verifier
- 결과가 실제로 맞는지 검증

Evaluator
- 품질 기준에 따라 평가

Failure Analyzer
- 실패 원인 분석

Refiner
- 프롬프트 또는 코드 개선

Memory Store
- 성공/실패 패턴 저장

Loop Controller
- 반복 여부와 종료 조건 관리

Human Gate
- 신규 기능, 방향 변경, 최종 종료 승인
```

## 9. 평가자와 검증자 분리 모델

루프 엔지니어링에서 가장 중요한 설계는 **생성자, 검증자, 평가자의 분리**입니다.

```text
Generator / Builder
- 산출물을 만든다.
- 프롬프트, 코드, 문서, 테스트 케이스를 생성한다.

Verifier
- 산출물이 실제로 동작하는지 확인한다.
- 가능한 한 실행 결과와 증거를 기준으로 판단한다.

Evaluator
- 산출물이 목표에 적합한지 평가한다.
- 품질, 완성도, 안정성, 유지보수성 등을 점수화한다.

Memory Manager
- 실패 원인, 성공 패턴, 개선 프롬프트를 저장한다.
- 다음 루프에서 참조 가능한 형태로 정리한다.

Human Gatekeeper
- 루프가 개발 범위를 벗어나지 않게 통제한다.
- 신규 기능, 큰 설계 변경, 종료 여부를 승인한다.
```

검증자는 “되는지 안 되는지”를 보는 역할이고,
평가자는 “목표에 비해 충분히 좋은지”를 보는 역할입니다.

## 10. 테스트 케이스 확장 방식

테스트 케이스는 처음부터 완벽하게 만들기 어렵습니다.
따라서 루프가 돌 때마다 테스트 케이스도 확장되어야 합니다.

### 10.1 기본 테스트

```text
- 정상 입력
- 정상 실행
- 정상 출력
- 필수 기능 작동 여부
```

### 10.2 실패 테스트

```text
- 잘못된 입력
- 누락된 값
- 빈 값
- 비정상 상태
- 권한 없음
- 네트워크 실패
```

### 10.3 엣지 케이스

```text
- 매우 긴 입력
- 매우 짧은 입력
- 특수 문자
- 다국어 입력
- 중복 데이터
- 경계값
```

### 10.4 회귀 테스트

```text
- 이전에 고친 문제가 다시 발생하지 않는지 확인
- 기존 기능이 깨지지 않았는지 확인
- 수정 전후 결과 비교
```

### 10.5 사용성 테스트

```text
- 사용자가 헷갈릴 수 있는 흐름
- 버튼/메뉴/에러 메시지의 명확성
- 실패 시 안내 문구
- 복구 가능한 동작 여부
```

## 11. 문제 기록 포맷

루프가 제대로 학습하려면 실패 기록이 표준화되어야 합니다.

```markdown
## 실패 기록

### 실패 ID
FAIL-001

### 발생 위치
로그인 폼 / 이메일 입력

### 테스트 케이스
잘못된 이메일 형식 입력

### 예상 결과
이메일 형식 오류 메시지가 표시되어야 함

### 실제 결과
아무 메시지 없이 제출 버튼이 비활성화됨

### 재현 절차
1. 로그인 페이지 접속
2. 이메일 입력란에 `abc` 입력
3. 비밀번호 입력
4. 제출 버튼 확인

### 원인 추정
이메일 검증은 수행되지만 사용자에게 오류 메시지를 표시하지 않음

### 심각도
중간

### 수정 방향
검증 실패 시 명확한 오류 메시지 표시

### 재테스트 결과
수정 후 PASS

### 다음 루프 참조 메모리
폼 검증 실패 시 버튼 비활성화만 하지 말고 반드시 사용자 안내 메시지를 표시할 것
```

## 12. 메모리 저장 항목

루프가 돌고 나서 메모리에 남겨야 할 항목은 다음과 같습니다.

```text
성공 패턴:
- 어떤 방식이 효과적이었는가?
- 어떤 프롬프트가 좋은 결과를 냈는가?
- 어떤 테스트가 문제를 잘 찾아냈는가?

실패 패턴:
- 어떤 접근이 실패했는가?
- 어떤 조건에서 문제가 반복되었는가?
- 어떤 검증 기준이 부족했는가?

재사용 규칙:
- 다음 작업에도 적용할 수 있는 규칙
- 반복 실수를 방지하는 체크리스트
- 개선된 프롬프트 템플릿

금지/주의 항목:
- 다시 시도하지 말아야 할 방식
- 사람 승인 전에는 진행하면 안 되는 항목
- 자동 판단이 위험한 영역
```

## 13. 루프 엔지니어링 운영 규칙

```text
1. 목표는 사람이 정한다.
2. 프롬프트는 에이전트가 생성하게 한다.
3. 작성자와 검증자는 분리한다.
4. 가능하면 검증자와 평가자도 분리한다.
5. 테스트 케이스는 계속 확장한다.
6. 실패는 반드시 기록한다.
7. 실패 원인은 다음 루프에서 참조한다.
8. 신규 기능은 자동 구현하지 않는다.
9. 종료 조건을 명확히 둔다.
10. 루프 한 바퀴마다 남는 자산이 있어야 한다.
11. 반복할수록 시스템이 더 똑똑해져야 한다.
12. 범위가 커질 때는 반드시 사람 승인을 받는다.
```

## 14. 실제 적용 예시: 웹앱 자유개선 루프

```text
목표:
기존 웹앱의 안정성과 사용성을 자동으로 개선한다.

1단계: 현재 기능 목록 수집
- 로그인
- 게시글 작성
- 파일 업로드
- 검색
- 관리자 페이지

2단계: 기본 테스트 케이스 생성
- 정상 로그인
- 잘못된 로그인
- 게시글 작성
- 빈 제목 제출
- 파일 업로드 실패
- 검색 결과 없음

3단계: AI에게 테스트 케이스 확장 요청
- 긴 제목
- 특수 문자
- 모바일 화면
- 권한 없는 접근
- 중복 제출
- 네트워크 지연

4단계: 전체 테스트 실행
- 자동 테스트 가능 항목은 실행
- UI 확인 항목은 검증자 Agent가 확인
- 실패 항목 기록

5단계: 실패 하나씩 수정
- FAIL-001 수정
- 해당 테스트 재실행
- PASS 확인 후 다음 실패로 이동

6단계: 전체 회귀 테스트
- 기존 기능 깨짐 여부 확인
- 이전 실패 재발 여부 확인

7단계: 확장 테스트 재생성
- 수정된 앱 기준으로 새 테스트 생성
- 숨은 문제 탐색

8단계: 신규 기능 제안
- 예: 실패 메시지 개선
- 예: 업로드 진행률 표시
- 예: 검색 결과 없음 안내 개선

9단계: 휴먼 승인
- 단순 개선은 백로그 등록
- 승인된 기능만 다음 루프에서 구현
```

## 15. 한 문장 정의

**루프 엔지니어링이란, 에이전트가 목표를 기준으로 스스로 프롬프트를 만들고, 테스트하고, 실패를 기록하고, 개선하고, 검증하며, 그 경험을 메모리에 쌓아 다음 실행의 품질을 높이는 반복 개선 시스템이다.**

## 16. 핵심 요약

```text
프롬프트를 잘 쓰는 시대에서
프롬프트를 스스로 개선하는 루프를 설계하는 시대로 이동하고 있다.

중요한 것은 한 번의 결과가 아니라,
반복할수록 좋아지는 구조다.

좋은 루프는 다음을 남긴다.
- 테스트 케이스
- 실패 기록
- 개선 프롬프트
- 검증 기준
- 재사용 가능한 메모리
- 사람 승인 대기 중인 기능 후보

그리고 좋은 루프는 반드시 멈출 줄 알아야 한다.
```
