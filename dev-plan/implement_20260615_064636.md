# implement_20260615_064636.md — Frozen Rule Semantic Runner / Adversary Semantic Gate

작성 일시: `2026-06-15 06:46:36 KST`

이 문서는 자가 개선 루프의 **마지막 닫힘 고리**인 *"adversary → M2 → M4 → frozen rulepack → 다음 루프 semantic 고정 게이트"* 를 코어에 배선하기 위한 개발 계획이다. 즉 **frozen rule semantic runner**(얼린 규칙을 실제로 실행하는 러너)와 **adversary semantic gate**(다음 루프에서 그 규칙을 고정 게이트로 강제)를 구현한다.

> 과장 금지: 상태는 코드 검증 근거(file:line / 테스트)로만 적는다. 미신뢰 LLM 산출 코드를 호스트에서 실행하지 않는다(R1 격리 하드 전제).

## 개발 목적

현재 파이프라인은 adversary 제안을 정적 필터 → M2 격리 confirm → rulepack candidate → replay corpus → replay → **freeze(`frozen_rulepack.json` + `lock_hash`)** 까지 만든다. 그러나 **얼린 규칙이 다음 루프에서 실제 게이트로 실행되지 않는다.** 학습이 "기록"까지만 되고 "강제"되지 않아 루프가 닫히지 않는다.

이 작업의 목적: 얼린 규칙을 **다음 루프의 결정론 hard 게이트로 실행**해, "이번 루프에서 발견·확인·replay된 약점"이 **다음 루프부터 영구적으로 막히게** 한다. (§14 "학습이 안전하게 들어가는 자리"의 실행 부분.)

### 진짜 공백 (이번에 구현)

1. **실행 가능 규칙 모델 부재** — `RulepackRule = {id, hash}` (`rulepack-shadow.ts:19`). 규칙을 실행할 **semantic spec(테스트 본문/명령 + 기대 verdict)** 이 없다. `FrozenRulepack.rules`도 해시 참조뿐.
2. **semantic runner 부재** — 얼린 규칙을 candidate worktree에 대해 **실제 실행**해 pass/fail을 내는 코드가 없다. (replay는 *제안*을 검증하는 별도 harness이지, 다음 루프 게이트가 아님.)
3. **게이트 주입 부재** — `frozen_rulepack.json`을 로드해 규칙을 `EvalGate`처럼 `runGates`에 넣는 경로가 없다. `builtin:rulepack-lock`은 **검증만** 한다(`builtin.ts:130`).
4. **다음-루프 적용 경로 부재** — 루프 N의 frozen 팩이 루프 N+1의 eval 계약으로 흘러 게이트가 되는 배선이 없다.
5. **실행 시점 provenance 부재** — 실행되는 규칙 본문이 `lock_hash`에 묶인 내용과 동일함을 보장하는 검증이 없다(semantic drift 위험).

§17 로드맵도 이를 명시: *"M4 replay-corpus EXECUTION(룰 실행)은 rulepack-runner/policy.lock 인프라 의존. 미구현."*

## 개발 범위

- `RulepackRule`에 **content-addressed 실행 spec** 추가(테스트 본문 + 기대 verdict). `lock_hash`가 실행 내용을 덮도록.
- **semantic runner**: frozen 규칙을 candidate worktree에 대해 **R1 격리 실행** → 규칙별 결정론 pass/fail → 집계.
- **builtin 게이트**(`builtin:rulepack-semantic`)와 eval 설정 블록 + 스키마 + dispatch + `runGates` 통합 + decision reason.
- **다음-루프 적용**: 루프 N frozen 팩 → 루프 N+1 게이트. `improve`/`orchestrate`/CLI에 적용 경로.
- 실행 시점 **provenance 바인딩**(실행 내용 ↔ `lock_hash`) + 안전 불변식.

## 제외 범위 (Non-goals, §18 정렬)

- 검증자/평가자를 LLM으로 만들지 않는다(러너는 결정론).
- **현재 루프**의 통과 바를 자동으로 낮추지 않는다(append-only 강화만, 적용은 next-loop-only).
- hidden acceptance/adversary 원문을 Builder/Skill에 노출하지 않는다.
- 미신뢰 코드를 호스트에서 실행하지 않는다(R1 필수).
- policy/rulepack을 hot loop 중 변경하지 않는다.

## Phase 1. 실행 가능 규칙 모델 (content-addressed RuleSpec)

### 목표

규칙을 **실행 가능**하게 만들고, 그 실행 내용을 `lock_hash`가 덮게 한다(드리프트 차단).

### 구현 태스크

- [ ] `RulepackRule`(`rulepack-shadow.ts:19`)을 확장하거나 `RuleSpec`를 신설: `{ id, hash, spec: { kind: 'command_test', body?, target_path?, command, expect: 'fail_to_pass'|'pass_to_pass', network: 'none' } }`. `hash = sha256(canonical(spec))` 로 **내용 주소화**.
- [ ] freeze 경로(`sdk/src/rulepack-freeze.ts`, `rulepack-candidate.ts`)가 M2-confirm된 proposal의 **실행 spec**(본문+기대 verdict)을 규칙에 담도록. `FrozenRulepack.rules`가 `{id,hash}`가 아니라 spec 포함(또는 spec를 별도 영역 + 해시 참조).
- [ ] `lock_hash` 계산에 spec 포함(이미 rules가 들어가면 자동). 기존 `checkRulepackLock`이 새 형식에서도 통과하도록 호환 유지/버전 bump(`schema_version`).
- [ ] adversary proposal → RuleSpec 변환기(`ruleSpecForProposal`)를 candidate 빌더에 추가.

### 자체 테스트

- [ ] proposal → RuleSpec → 해시 안정성(동일 spec 동일 hash) 단위 테스트.
- [ ] freeze 산출 `frozen_rulepack.json`에 실행 spec이 포함되고 `lock_hash`가 spec 변조 시 바뀜.
- [ ] 기존 `checkRulepackLock`(무결성) 회귀 통과.

### 완료 조건

- [ ] frozen 팩이 **실행 가능 규칙**을 content-addressed로 담는다 + 무결성 게이트 회귀 통과.

## Phase 2. Frozen Rule Semantic Runner (R1 실행)

### 목표

얼린 규칙을 candidate worktree에 대해 **격리 실행**해 규칙별 결정론 verdict를 낸다.

## 안전 불변식 (요약)

1. 미신뢰 규칙은 **R1 격리(network=none)** 에서만 실행. 런타임 부재 → fail-closed.
2. 실행 내용 = `lock_hash`에 묶인 spec(해시 일치). 불일치 → 미실행·reject.
3. **next-loop-only**: producing loop ≠ 현재 loop. 자가 적용 금지.
4. **append-only·replaySafe** 인 frozen 팩만 실행(실행 직전 재확인).
5. 규칙은 **결정론**(exit-code)만. 비결정/네트워크/시간 의존 금지.
6. hidden/secret 비노출(본문·출력 스캔).
7. 현재 루프 통과 바를 낮추지 않음(강화 방향, 다음 루프부터).
