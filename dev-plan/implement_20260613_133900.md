## 개발 목적

[implement_20260613_130927.md](./implement_20260613_130927.md) Phase 0 영역1 조사 결론(분리 구현)에 따라, **평가자(Evaluator)/Critic 분리**를 advisory only로 구현한다. 비전 §4·9의 "검증자/평가자 분리"를 충족하되, 최종 accept 판정은 계속 deterministic decision engine의 `ALL_PASS`만 담당한다.

핵심 목표는 다음 2개다.

1. 독립 provider로 실행되는 advisory critic/evaluator runner를 도입하고, 품질 점수/지적을 eval-report `advisoryFindings`에만 기록한다.
2. `resolveSameModelReview`를 adapter provider identity 비교로 승격하고, 불명이면 보수적 true를 유지한다.

## 구현 중 발견(gap-only 재확정)

구현 착수 시 코드 재확인 결과: **evaluator/critic 실행 메커니즘은 이미 존재한다.** advisory는 `gate.type==='advisory'` 게이트로서 `executeCommandGate`로 실행되며(사용자가 `command:`/codex로 임의 reviewer를 advisory 게이트로 지정 가능), 그 결과는 `advisoryFindingsFor()`가 decision 불참 상태로 eval-report에 기록한다. 따라서 **새 evaluator adapter를 추가하면 advisory command gate와 기능이 중복된다.**

→ 본 워크스트림의 실제 gap은 **provider-identity 기반 `same_model_review` 승격**(목표 2)으로 좁힌다. 목표 1의 "독립 LLM evaluator runner 신규 adapter"는 실제 2nd provider 통합(예: Anthropic) 시점의 후속으로 분리하고, 본 워크스트림에서는 advisory finding에 `builder_provider`/`reviewer_provider`를 노출하는 관측성 보강만 수행한다.

## 개발 범위

- `packages/agent-adapters`에 advisory critic/evaluator adapter 추가(provider id 노출)
- advisory 게이트 결과에 evaluator 품질 점수/지적을 advisory 블록으로 기록
- `resolveSameModelReview`를 builder provider ↔ reviewer provider 비교로 승격
- evaluator 실패 시 advisory-only 흡수(required 아님 → decision 불참)
- evaluator 출력 redaction(hidden/token/원문 발견 입력 비노출)
- decision 14규칙·accept 경로 불변 회귀 테스트

## 문서 기반 제약 사항

- 최종 accept 판정은 deterministic decision engine의 `ALL_PASS`만 허용한다.
- evaluator/critic는 advisory only이며 decision 14규칙에 참여하지 않는다.
- `same_model_review`는 "reviewer provider independence not guaranteed" 표시이며, 불명이면 보수적 true.
- evaluator 출력에 hidden acceptance·token·원문 발견 입력을 노출하지 않는다.
