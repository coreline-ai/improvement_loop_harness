# implement_20260617_211537.md

작성 일시: `2026-06-17 21:15:37 KST`

이 문서는 **Product-100 Codex Live UAT** 개발 범위와 테스트 환경 구축 조건을 고정한다. 목적은 기존 R13/R15/R16 controlled/live 증거 다음 단계로, 실제 Codex가 수정하고, 별도 Codex adversary reviewer가 깨는 테스트를 제안하며, 그 테스트가 M2/M4/replay/freeze를 거쳐 다음 루프의 고정 semantic gate로 적용되는지까지 검증하는 풀 테스트 시나리오를 구현하는 것이다.

> 과장 금지: 이 문서는 `제품 전체 100% PASS`를 선언하는 문서가 아니다. `PRODUCT_100_CODEX_LIVE_PASS`는 아래 PASS 공식과 증거 계약을 모두 만족할 때만 출력한다. 중간 단계 PASS는 반드시 `controlled`, `representative`, `blocked`, `verification-only` 범위를 붙인다.

## 개발 목적

현재 확인된 미완 항목을 하나의 재현 가능한 풀 UAT로 묶는다.

| 미완 항목 | 현재 상태 | Product-100에서 닫는 방식 |
| --- | --- | --- |
| hidden/adversary eval 자동생성 | 명시 hidden test와 rulepack semantic gate는 가능하나 자동 생성은 후속 | seeded issue metadata에서 hidden acceptance + adversary proposal 후보를 자동 생성하고 agent에는 미노출 |
| real Codex adversary reviewer live | controlled command adversary는 가능, real reviewer provenance는 후속 | 별도 Codex reviewer command를 호출하고 fixed prompt hash/version, accepted proposal, `same_model_review=false`를 ledger에 기록 |
| 더 큰 실제 서비스/프레임워크 corpus | repo-matrix controlled 16셀, 대표 Python/monorepo live 2셀 | Node monorepo, Python service, React/Next-like, CLI, security fixture를 하나의 Product-100 corpus로 실행 |
| 실제 Codex 수정 포함 | R13/R15/R16에서 확인됨 | 모든 Product-100 issue는 real Codex builder/challenger가 후보를 만들고, command fixture만으로는 PASS 금지 |
| 최고 수정 선택 + 순차 수정 | R13 controlled RU-3 strict-best PASS | 각 repo/issue에서 `strict_score_improvement=true`와 rediscovery-after-fix를 필수화 |

## 개발 범위

- Product-100 UAT 드라이버: `scripts/uat/product-100-codex-live-uat.mjs`
- package script: `uat:product-100:codex-live`
- Product-100 corpus 생성기: 실제 서비스/프레임워크에 가까운 5개 controlled repo fixture를 만들고 각 repo에 2개 이상 seeded bug를 주입
- hidden/adversary eval generator: issue metadata에서 hidden acceptance source와 adversary proposal seed를 만들되, Builder/Challenger 입력에는 숨김
- real Codex Builder/Challenger: candidate 생성은 `codex`/ChatGPT OAuth proxy 경로로만 인정
- real Codex adversary reviewer lane: `VIBELOOP_ADVERSARY_REVIEWER_COMMAND` 또는 전용 wrapper를 통해 별도 Codex reviewer 실행
- M2/M4/frozen semantic next-loop: accepted reviewer proposal을 R1 `network=none`에서 confirm/replay/freeze하고 다음 루프 `builtin:rulepack-semantic` required gate로 실행
- GitHub draft PR: 각 repo/issue별 PR 후보를 draft PR로만 생성, auto-merge 금지
- evidence/audit: durable evidence bundle + manifest + ledger + PR URLs + proxy stats + release audit까지 검증

## 제외 범위

- 실제 고객 production repo에 직접 patch를 적용하지 않는다. 모든 live repo는 throwaway/private GitHub repo 또는 operator가 명시한 test repo만 사용한다.
- auto-merge는 영구 금지한다.
- LLM/adversary 판단을 current-loop accept gate로 승격하지 않는다. adversary 제안은 M2/M4/freeze 후 **다음 루프 고정 semantic gate**로만 반영한다.
- Docker/R1 또는 Postgres가 blocked인 환경에서 `PRODUCT_100_CODEX_LIVE_PASS`를 출력하지 않는다. `PRODUCT_100_CODEX_LIVE_BLOCKED`로 끝낸다.
- fixture-only, command-only, single-repo representative 결과를 제품 전체 100% PASS로 부르지 않는다.
