1. `loop_engineering_notes.md` 부록 C의 검토 질문을 실제 코드와 대조해 발견 기록을 채운다.

3. deterministic decision engine의 `ALL_PASS` 단일 accept 경로를 약화시키는 설계를 사전에 차단한다.

## 문서 기반 제약 사항

- 최종 accept 판정은 deterministic decision engine의 `ALL_PASS`만 허용한다.
- evaluator/critic/learning/convergence는 advisory 또는 orchestrator stop 신호일 뿐 decision에 참여하지 않는다.
- learning/memory는 읽기 참조·우선순위 힌트로만 사용한다.
- 발견 입력·learning 자산은 untrusted로 취급하고 원문 텍스트 주입을 금지한다.
- hidden acceptance 원문은 agent/Skill/learning/artifact log에 노출하지 않는다.
- 새 reason code가 필요하면 `EVAL_ENGINE_SPEC`, `decision/rules.ts`, `eval-report.schema.json`, fixture를 동시에 바꾸는 별도 계획을 만든다.

### 발견 기록

#### 영역1 evaluator — 결론: 분리 구현

- **확장점 존재**: `packages/sdk/src/run.ts`의 `advisoryFindingsFor()`가 `gate.type==='advisory'` 게이트를 모아 `{source, gate, status, authority:'advisory', same_model_review}`로 eval-report에 기록한다. `resolveSameModelReview(agentSpec, criticConfig)`는 strict(`require_different_provider`)→false, `mock:`→false, 그 외→**보수적 true**. eval `critic` config(`require_different_provider`)도 이미 존재.
- **빠진 것**: advisory 게이트는 `executeCommandGate`로 실행되는 일반 명령 게이트일 뿐, **독립 모델을 실제 실행하는 critic/evaluator runner가 없다**. `same_model_review`는 provider 동일성을 확증하지 못해 항상 보수적 true.
- **Q3 노출 위치**: eval-report `advisoryFindings` 블록이 이미 존재 → 품질 점수는 여기(또는 별도 advisory artifact)에 기록하면 decision 불참 유지.
- **Q4 승격 최소 계약**: adapter가 provider id를 노출 → builder agentSpec provider와 비교. 불명이면 보수적 true 유지.
- decision 14규칙은 advisory를 참조하지 않으므로(`decision/engine.ts`) evaluator 추가는 신뢰 경계 불변.

#### 영역2 learning/failure ledger — 결론: 분리 구현(스키마 보강 포함)

- **dead schema 확인**: Prisma `Learning`(projectId/sourceLoopId/status/title/body/evidenceRefs/reviewedAt)은 `apps/server/src/store-contract.test.ts`에서만 참조 — 루프 종료 시 기록하는 production 경로도, candidate 우선순위에서 읽는 경로도 없다. `SkillVersion`은 레지스트리 용도.
- **Q2 안전한 주입 지점 확정**: `ImprovementCandidate.priority`(Int) 필드가 이미 존재 → learning 기반 **우선순위 가중**이 안전한 참조 지점. task objective/agent 프롬프트 주입은 금지(injection).
- **Q3 deterministic 군집 키**: 현 fingerprint = `sha256(source+위치+유형)`. 이를 일반화한 `(source, riskAreaHint, errorCode)` 군집 키로 LLM 없이 1차 군집화 가능.
- **Q4 ledger 포맷**: §11 실패 기록 포맷을 `Learning.body`(원문 텍스트 미저장, 구조화 요약)로 재사용.
- 필드 보강 필요 가능성 → Prisma migration 동반 → 별도 계획.

#### 영역3 test expansion — 결론: 회귀 고정만(별도 단계 도입 보류)

- **이미 부분 충족**: builder가 write_scope 내 테스트 파일을 생성하고, `test-integrity.ts`가 assertion 삭제·`test.skip`·forbidden 패턴을 차단하며, `test-on-base.ts`가 `task.acceptance.required_tests`에 fail-on-base→pass-on-candidate를 강제한다.
- **anti-gaming 경로 존재**: 별도 확장 단계 없이도 test-on-base가 "base에서도 통과하는 무의미 테스트"를 차단.
- **결론**: 별도 test-expansion 단계는 **현 시점 불필요**. builder 산출물 + test-on-base 강제가 충분. 추후 필요 시 hidden acceptance(§7.3) 경로 재사용. 본 워크스트림에서는 "확장 테스트도 기존 guard/decision/ test-on-base를 우회 못 함" 회귀 고정만 고려.

#### 영역4 convergence — 결론: 분리 구현

- **Q1 삽입 지점**: `apps/server/src/orchestrator/guardrails.ts`의 `evaluateBeforeLoop`가 종료/정지 판단의 단일 지점. 여기에 "신규 수확 없음 N회" 카운터를 추가.
- **Q2 충돌 없음**: 수렴 "종료"는 orchestrator **정지**일 뿐 accept를 만들지 않음 → 절대 제약과 무충돌.
- **Q3 사람 승인**: 기존 `createApproval` + `orchestrator.*` 이벤트 경로 재사용(`scheduler.ts`의 needs_human_review 흐름과 동일 패턴).
