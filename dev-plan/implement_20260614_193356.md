# implement_20260614_193356.md

작성 일시: `2026-06-14 19:33:56 KST`

이 문서는 `vibeloop-harness` Skill을 실제 사용자 시나리오에 맞게 재정렬하기 위한 **진행 중 개발 계획 초안**이다. 사용자가 계속 내용을 추가할 예정이므로, 본 문서는 최종 고정본이 아니라 이후 항목을 계속 누적·정정하는 작업 문서로 유지한다.

## 개발 목적

현재 fixture 기반 deterministic UAT와 실제 사용자 LLM/Codex 사용 시나리오를 분리하고, Skill이 다음 두 입력 모드를 기본 지원하도록 개발 범위를 정리한다.

1. **사용자 지정 문제 모드**: 사용자가 특정 문제를 제시하면 그 문제 1개만 task/eval로 변환하고 수정 루프를 실행한다.
2. **자동 문제 발견 모드**: 사용자가 “자율 개선”만 요청하면 Skill이 repo를 스캔해 문제 후보를 만들고, 우선순위 1개를 선택해 task/eval로 변환한 뒤 수정 루프를 실행한다.

핵심 목표는 “LLM이 후보를 만들고, Harness가 고정 Verifier/Evaluator/Arbiter로만 판정한다”는 제품 동작을 실제 사용자 흐름에 맞게 만드는 것이다.

## 개발 범위

- 사용자 지정 문제 모드 정의
- 자동 문제 발견 모드 정의
- 문제 후보 생성/우선순위/선택 기준 정의
- 선택된 문제 1개를 task/eval로 변환하는 플로우 정의
- Builder / Challenger / Adversary 후보 생성 플로우와 연결
- Verifier / Evaluator / Arbiter 판정 기준 연결
- PR 후보 생성 조건 재정의
- fixture baseline UAT와 real Codex/LLM live UAT 명칭 분리
- 계속 추가될 사용자 요구사항을 후속 Phase로 누적

## 제외 범위

- 이 문서는 전체 제품 완료 선언을 하지 않는다. 각 구현 항목은 증거 범위별로 `완료/부분/미구현`을 분리한다.
- fixture 기반 PASS를 실사용자 PASS로 부르지 않는다.
- 평가/검증 기준 자동 개선 루프 구현은 아직 포함하지 않는다. 별도 요구가 확정되면 Phase로 추가한다.
- production-grade 선언은 RU-3 full loop와 Adversary/rulepack freeze까지 검증되기 전까지 하지 않는다.

## 전문가 리뷰 결론 — 우선 구현 순서

### 총평

현재 추가된 개발 내용은 프로젝트 목적에 맞다. 다만 **구현 순서는 재배치해야 한다.** 지금 상태에서 advisory reviewer나 option 2 rulepack 승격을 먼저 구현하면 “자율 개선처럼 보이는 기능”은 생기지만, selected patch/provenance/live LLM 증거가 약해 실사용 신뢰성이 올라가지 않는다.

따라서 구현 순서는 아래처럼 고정한다.

```text
P0 문서/용어 과장 방지
→ P1 trust-floor 보강
→ P2 입력/문제 선택 계약
→ P3 사용자 지정 문제 모드
→ P4 fixture baseline 명칭 정정
→ P5 실제 Codex/LLM live UAT
→ P6 자동 문제 발견 모드
→ P7 분리 context advisory reviewer
→ P8 option 2 안전 승격(rulepack freeze) 후속
```

### RU-3 격상 결정

| 결정           | 내용                                                                                                                                                                                                                                    |
| -------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 현재 결정      | **현 `orchestrate --promote-branch`는 local RU-3 substrate로 격상. Full/live RU-3로는 격상하지 않는다.**                                                                                                                                                                                          |
| 이유           | 현재는 discover→task 생성→issue별 `runImprovementLoop`까지의 core substrate다. 단일 이슈 local branch는 `improve --promote-branch`로 가능하지만, 선택 patch 누적 적용·수정 후 재탐색·local orchestrate branch는 `--promote-branch`로 가능하다. 단 GitHub draft PR/push, live RU-3 UAT, hidden/adversary/policy-rich eval은 없다. |
| RU-3 승격 조건 | `apply selected patch → commit/integration branch → rediscover → next issue → core PR branch → GitHub draft PR 후보`가 한 명령에서 evidence로 남아야 한다.                                                                                       |
| live UAT 조건  | fixture가 아니라 실 Codex + GitHub repo + draft PR/branch evidence로 `verification_status=pass`를 남겨야 한다.                                                                                                                          |
| 금지 표현      | 현 상태를 `실사용자 자동 문제 발견 full loop PASS` 또는 `풀 자율개선 PASS`로 부르지 않는다.                                                                                                                                             |

## Phase 7. 분리 컨텍스트 적대적 advisory 리뷰어

### 목표

별도 provider/fresh context의 CLI를 **accept 게이트가 아닌 advisory reviewer**로 붙여, LLM이 테스트가 못 잡는 결함·엣지·회귀 가능성을 지적하되 최종 판정은 절대 바꾸지 않도록 한다.

### 채택/폐기 결정

| 옵션                                                        | 결정 | 이유                                                                                             |
| ----------------------------------------------------------- | ---- | ------------------------------------------------------------------------------------------------ |
| 옵션 1 — 분리 CLI 적대적 advisory 리뷰어                    | 채택 | 결정론 accept 기준을 깨지 않고, hidden/builder transcript 미노출 상태에서 PR 전 검토 신호를 제공 |
| 옵션 2 — adversary LLM 테스트 생성→격리 실행→ephemeral gate | 보류 | 가치가 크지만 run마다 합격선이 흔들릴 수 있음. rulepack freeze/M2/M4 계약 확정 후 별도 트랙      |
| 옵션 3 — 분리 CLI verdict를 accept 게이트로 승격            | 폐기 | LLM 의견이 accept에 들어가면 비결정·프롬프트 인젝션·self-review dilution 위험이 게이트로 유입    |

### 옵션 2 안전 승격 경로

옵션 2는 폐기가 아니라 **후속 고가치 트랙**이다. 단, Adversary LLM이 매 실행마다 새 테스트를 만들면 accept 기준이 흔들리므로 현재 루프의 즉시 accept 게이트로 사용하지 않는다.

```text
advisory/adversary finding
→ 검증된 테스트 후보
→ M2: deterministic filter + R1 격리 실행
→ M4: replay corpus로 오탐/과잉 제약 확인
→ rulepack freeze(hash 고정, append-only)
→ 다음 루프의 고정 Verifier/Evaluator 게이트로 편입
```

| 단계               | 목적                                      | accept 영향                                             |
| ------------------ | ----------------------------------------- | ------------------------------------------------------- |
| advisory/adversary | LLM이 깨질 가능성·엣지·회귀 테스트를 제안 | 현재 루프 accept 변경 금지                              |
| 테스트 후보 저장   | 제안 내용을 구조화 산출물로 보존          | accept 영향 없음                                        |
| M2 필터/격리 실행  | scope/no weakening/no hidden leak 확인    | 현재 루프에서는 candidate risk/advisory 신호까지만 허용 |
| M4 replay          | 기존 corpus에서 오탐·과잉 제약 여부 확인  | replay-safe가 아니면 폐기                               |
| rulepack freeze    | hash가 고정된 append-only rule로 승격     | **다음 루프부터** deterministic gate로 적용             |
| 다음 루프 적용     | 얼린 rulepack을 고정 Verifier/Evaluator화 | 이 시점부터 accept 공식에 포함 가능                     |

불변식:

- [ ] LLM이 즉석 생성한 테스트는 현재 루프의 최종 accept를 직접 바꾸지 않는다.
- [ ] 현재 루프 중 `rulepack`/`policy.lock`을 완화하거나 동적으로 바꾸지 않는다.
- [ ] freeze된 rulepack만 다음 루프의 고정 게이트가 될 수 있다.
- [ ] rulepack 승격은 append-only이며 hash/provenance에 기록한다.
- [ ] hidden acceptance 원문·경로·시크릿이 adversary/advisory 입력이나 generated test에 섞이면 폐기한다.

### 구현 태스크

- [ ] `eval.yaml`에 advisory reviewer gate 예시를 추가한다.
- [ ] `critic.reviewer_provider` / `critic.require_different_provider` 사용 기준을 문서화한다.
- [ ] advisory reviewer 입력을 `diff + task/eval acceptance 기준`으로 제한한다.
- [ ] builder transcript, hidden acceptance test, hidden sentinel, OAuth/token 정보를 advisory reviewer에 노출하지 않는다.
- [ ] prompt는 적대적으로 고정한다: “이 변경을 통과시키지 말고 깨라 — 테스트가 못 잡는 결함/엣지/회귀를 찾아라.”
- [ ] advisory 출력 JSON 계약을 정의한다: `severity`, `findings[]`, `suggested_tests[]`, `confidence`, `requires_human_review_signal`.
- [ ] advisory 결과를 `advisory_findings`로만 저장한다.
- [ ] `decision`, `qualified`, `selected_candidate_id`, PR candidate predicate는 advisory 결과로 절대 변경하지 않는다.
- [ ] summarizer에 `advisoryReviewRecommended` 또는 `review_advisory_before_pr` 신호를 추가한다.
- [ ] PR body/report에는 advisory findings를 표시하되, deterministic accept와 구분해 표기한다.
- [ ] 옵션 2 경로는 `advisory → 테스트 후보 → M2/M4 → rulepack freeze → 다음 루프 고정 게이트`로만 문서화하고, 현재 루프 즉시 accept 게이트로 연결하지 않는다.

### 자체 테스트

- [ ] advisory gate가 fail/nonzero여도 deterministic `decision=accept`를 reject로 바꾸지 않는지 확인
- [ ] advisory finding이 있어도 `qualified`와 `selected_candidate_id`가 바뀌지 않는지 확인
- [ ] summarizer가 PR 후보를 유지하되 advisory review 신호를 함께 출력하는지 확인
