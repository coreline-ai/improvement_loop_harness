## 개발 목적

`artifact-leak` core gate는 **agent stdout/stderr**의 context/secret/hidden marker 누설을 커널 decision에서 `reject / GUARD_ARTIFACT_LEAK`로 차단하는 1차 목표를 달성했다. 이번 후속 개발의 목적은 대규모 DLP 기능 확장이 아니라, 다음 세 가지를 v1.1에서 정확히 마무리하는 것이다.

```text
v1.1 즉시 보강
  → 불필요한 token_like reject 제거
  → v1 커버 범위(agent stdout/stderr 한정)와 제외 artifact 명문화
  → false-positive를 만들 수 있는 token_like reject 사용을 기본/fixture 경로에서 제거
  → configured artifact-leak의 not-evaluated pass fallback 방지

v2 별도 개발로 분리
  → candidate.patch secret/marker scan
  → project gate stdout/stderr redact-only
  → always-on redact-only baseline compatibility 검토
```

핵심 원칙:

```text
v1.1 = 신뢰 경계 설명 정정 + 오탐 표면 축소 + 기존 gate 약화 방지
v2   = 새 누설면 구현 확장(candidate.patch, gate logs, always-on baseline)
```

## 개발 범위

- L1: project command gate stdout/stderr가 v1 artifact-leak scan/redaction 대상이 아님을 문서와 UAT 설명에 명확히 기록
- L2: Skill loop SKU fixture의 불필요한 `artifact_leak.builtins.token_like: true` 제거
- L3: `token_like` false-positive 위험을 줄이기 위해 v1.1에서는 기본/fixture 경로에서 token-like reject를 켜지 않는 정책으로 고정
- L4: patch 내 secret/marker scan을 v2 우선순위 1순위로 문서화
- L5: `artifact_leak` 미설정 프로젝트에 대한 always-on redact-only baseline은 즉시 구현하지 않고 후속 compatibility 과제로 문서화
- L6: `input/eval.yaml` 같은 원본 정책 artifact에는 provenance 목적상 raw config가 보관될 수 있음을 명문화
- L7: `artifact_leak`가 설정된 normal agent run에서 `builtin:artifact-leak`가 `not evaluated` pass로 fail-open 되지 않도록 구현/테스트 기준 고정
- 변경 후 positive/adversarial UAT와 기존 회귀 테스트 재검증

## 제외 범위

- 이번 v1.1에서 project gate stdout/stderr redaction 구현은 제외한다. 단, v2 Phase로 상세 계획화한다.
- 이번 v1.1에서 candidate patch secret/marker scan 구현은 제외한다. 단, v2 최우선 과제로 고정한다.
- 이번 v1.1에서 `token_like` detector를 confidence class별로 재설계하지 않는다. `secret/password` assignment의 reject 정책 분리는 v2에서 별도 설계한다.
- 이번 v1.1에서 `input/eval.yaml` 원본 config를 redact 저장하지 않는다. provenance/hash 재현성 영향이 있으므로 별도 설계가 필요하다.
- 이번 v1.1에서 `retry_eval_only`를 agent output scan 대상으로 확장하지 않는다. 단, safe skipped-agent output을 명시적으로 precomputed artifact-leak result로 전달해 `not evaluated` fallback에 의존하지 않게 한다.
- hidden acceptance 원문 자동 fingerprint/semantic similarity 탐지는 제외한다.
- LLM 판단 기반 누설 탐지는 제외한다.
- 자동 merge 또는 PR 자동 승인 기능은 제외한다.
- 기존 하위호환 결정인 `artifact_leak` config-gated 동작을 즉시 always-on으로 바꾸지 않는다.
- 이번 v1.1에서 `skills/vibeloop-harness/SKILL.md`에 artifact-leak 커버리지 캐비엣을 추가하지 않는다. SKILL.md는 배포되는 제품 표면이므로 thin wrapper로 유지하고(직전 정리 커밋 `cb58e3c`), 커버 범위 상세는 번들과 함께 가는 `references/usage.md` 또는 배포되지 않는 내부 `docs/*`에만 둔다.
- 커버리지 한계 문구를 여러 문서에 분산 복제하지 않는다. 권위 있는 1곳(`docs/EVAL_ENGINE_SPEC.md`)에 표로 두고, 다른 문서는 1줄 잔여리스크 + 그곳을 참조한다.

## 문서 기반 제약 사항

- `ALL_PASS`는 계속 correctness pass의 의미로 유지한다.
- `GUARD_ARTIFACT_LEAK`는 verifier guard 계열이며 `ALL_PASS` 전에 reject한다.
- v1 artifact-leak의 확정 커버리지는 **agent stdout/stderr**다.
- project command gate stdout/stderr, candidate patch, input/eval.yaml 원본 config는 v1.1에서 과대 홍보하지 않는다.
- redaction과 reject는 분리한다. redaction은 넓게, reject는 정밀하게 적용한다.
- false positive로 정상 candidate를 막는 설정은 fixture/기본 경로에서 피한다.
- `forbidden_literals.value`에는 실제 운영 secret이 아니라 UAT sentinel, 이전 task id, 정책상 금지된 context marker처럼 회수 가능한 값을 넣는 것을 권장한다.
- `artifact_leak` config가 있는데 precomputed 결과가 없으면 `builtin:artifact-leak`는 `pass`가 아니라 `fail`(또는 에러)로 처리한다. config가 없으면 기존대로 not-configured pass(하위호환). normal run은 결과를 항상 전달하므로 이미 fail-open이 아니고, `retry_eval_only`는 명시적 result 전달로 불변식을 충족한다.
- `retry_eval_only`는 agent를 재실행하지 않으므로 safe skipped-agent output에 대한 명시적 scan result를 넘기는 방식으로 예외 처리한다.

## Phase 4. L7 not-evaluated fail-open 방지

### 목표

코드 확인(2026-06-14): normal agent run은 [run.ts:687](../packages/sdk/src/run.ts#L687)이 항상 `artifactLeakResult`를 계산·전달하므로 **현재 fail-open이 아니다**(활성 구멍 없음). 유일한 미평가 경로는 `retry_eval_only`([run.ts:626](../packages/sdk/src/run.ts#L626))이며, 이때 agent는 실행되지 않고 stdout은 안전 정적 문자열이라 누설할 agent 출력 자체가 없다. 그러나 [builtin.ts:83](../packages/eval-engine/src/builtin.ts#L83)의 `context.artifactLeak ?? {status:'pass', ...}`는 잠재 footgun이다.

따라서 목표는 활성 버그 수정이 아니라 **잠재 footgun을 시끄러운 불변식으로 고정**하는 것이다: `artifact_leak` config가 있는데 precomputed 결과가 없으면 `builtin:artifact-leak`는 `pass`가 아니라 **명시적으로 fail(또는 에러)** 한다. `retry_eval_only`는 safe skipped-agent output에 대한 **명시적 scan result를 전달**해 이 불변식을 충족한다(undefined→pass에 의존하지 않음).

### 구현 태스크

- [ ] `builtin:artifact-leak`가 "config 설정됨 + 결과 없음"을 구분하도록 gate context에 설정 여부(또는 `evalConfig.artifact_leak` 참조)를 전달한다 — 현재 builtin은 config 인지가 없어 무조건 pass fallback함
- [ ] 위 경우 `builtin:artifact-leak`는 `pass`가 아니라 `fail`(또는 에러)을 반환하도록 [builtin.ts:83](../packages/eval-engine/src/builtin.ts#L83)의 `?? pass` 폴백을 교체
- [ ] `retry_eval_only` 경로는 raw agent output이 없음을 문서화하고, static skipped-agent stdout/stderr에 대한 명시적 scan result를 gate context에 전달([run.ts:625](../packages/sdk/src/run.ts#L625)에서 `artifactLeakResult`를 undefined로 두지 않음)
- [ ] 단위 테스트: config 있음+결과 없음 → fail, config 없음 → 기존대로 not-configured pass(하위호환 유지)
- [ ] grep: `artifact-leak not evaluated`가 정상 configured run의 gate log/summary에 나오지 않음

### 자체 테스트

- [ ] `pnpm test -- packages/eval-engine/src/builtin.test.ts packages/sdk/src/run*.test.ts` 또는 관련 단위 테스트

## 잔여 리스크 / 후속 과제

- project command gate stdout/stderr redaction은 v2에서 구현 전까지 잔여 누설면이다.
- candidate patch secret/marker scan은 PR 안전성과 직접 관련되므로 v2 최우선으로 다룬다.
- `input/eval.yaml` 원본 config에는 raw forbidden literal 값이 남을 수 있으므로 실제 운영 secret을 literal로 넣지 않는 가이드가 필요하다.
- `artifact_leak` 미설정 프로젝트 보호 0 문제는 always-on redact-only compatibility 검토 후 결정한다.
- token-like detector의 high-confidence/low-confidence 분리는 v2에서 별도 설계한다.
- 과거 완료 계획 파일은 변경 이력 보존을 위해 수정하지 않고, v2 계획에서 링크만 참조한다.
