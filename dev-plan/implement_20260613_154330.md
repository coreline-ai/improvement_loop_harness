## 개발 목적

VibeLoop Harness에 **같은 문제 내부 개선 루프(Same-Issue Refinement Loop)**를 1급 루틴으로 추가한다.

현재 루프는 `문제 발견 → 수정 시도 → deterministic 검증 → 통과 시 PR 후보` 흐름은 명확하지만, 통과한 수정안이 **같은 문제 범위 안에서 더 나은 방향으로 개선될 수 있는지 판단하고, 더 좋은 후보를 재수정·재검증·선택하는 중간 판단 과정**이 별도 단계로 정의되어 있지 않다.

이 개발의 목표는 다음이다.

1. 통과한 후보를 즉시 PR로 보내기 전에 `RefinementJudge`가 개선 여지를 판단한다.
2. 개선 여지가 있으면 **동일 문제·동일 acceptance 범위 안에서만** 제한된 refinement attempt를 수행한다.
3. 모든 후보는 기존 deterministic gate를 다시 통과해야 하며, 최종 PR 후보는 **통과한 후보들 중 비교 기준상 가장 좋은 후보**로 선택한다.
4. LLM/evaluator의 판단은 advisory로만 사용하고, accept 권한은 기존 decision engine의 `ALL_PASS`에만 둔다.

## 문서 기반 제약 사항

- 최종 accept 판정은 deterministic decision engine의 `ALL_PASS`만 허용한다.
- refinement는 **통과한 후보를 더 좋게 만들 수 있는지 판단하는 후보 선택 루프**이지, 실패 후보를 임의로 통과시키는 루프가 아니다.
- refinement는 **같은 문제, 같은 objective, 같은 acceptance, 같은 write_scope** 안에서만 허용한다.
- refinement 중 eval/test/hidden acceptance/protected path 변경은 즉시 reject 또는 human review다.
- 이전 accepted 후보는 보존한다. 새 refinement 후보가 실패하면 이전 accepted 후보를 폐기하지 않는다.
- 후보 비교는 accepted 후보끼리만 한다. 실패 후보는 비교 대상이 아니다.
- `RefinementJudge`는 advisory다. 개선 필요성 제안은 가능하지만 gate 판정을 바꾸지 않는다.

## 핵심 루틴

```text
문제 발견
  ↓
수정 attempt-1
  ↓
deterministic 검증
  ├─ 실패 → 실패 분석/동일 문제 재시도 또는 중단
  └─ 통과 → accepted_candidate 저장
              ↓
            RefinementJudge
              ├─ 개선 여지 없음 → PR 후보 생성
              ├─ 같은 범위 개선 가능 → refinement attempt-2
              └─ 범위 확장/위험 증가 → human review 또는 기존 후보 유지

refinement attempt-2
  ↓
deterministic 재검증
  ├─ 실패 → attempt-1 유지
  └─ 통과 → attempt-1 vs attempt-2 비교
              ↓
            best accepted candidate 선택
              ↓
            PR 후보 생성
```

## RefinementJudge 계약

### 입력

- 원래 candidate/task objective/write_scope/acceptance
- accepted candidate의 diff, changed files, eval-report, evidence summary
- required/public/hidden gate 결과 요약(원문 hidden test 노출 금지)
- advisory findings(`same_model_review`, reviewer provider, critic/advisory gate 상태)
- diff risk 지표(변경 파일 수, protected path 접촉, 테스트 변경 여부, 삭제량)
- 실패·학습 원장 힌트가 있는 경우 구조화 요약만 참조

### 출력

| 출력                     | 의미                                                                  | 후속 동작                       |
| ------------------------ | --------------------------------------------------------------------- | ------------------------------- |
| `stop_and_pr`            | 현재 accepted candidate가 충분히 좋음                                 | PR 후보 생성                    |
| `refine_same_issue`      | 같은 문제·같은 acceptance 안에서 더 나은 후보를 만들 여지가 있음      | bounded refinement attempt 실행 |
| `keep_previous`          | 새 후보가 통과했지만 이전 후보보다 개선 근거가 약함                   | 이전 accepted candidate 유지    |
| `needs_human_review`     | 범위 확장, 위험 증가, 기준 변경 요구, 품질 판단 불확실성이 큼         | approval queue 또는 수동 검토   |
| `reject_refinement_path` | eval/test/protected path 변경 등 refinement 자체가 신뢰 경계를 위반함 | refinement 폐기, 이전 후보 유지 |

### refinement hint 생성 규칙

- 원래 objective를 다시 쓰지 않는다.
- hidden acceptance 원문을 절대 노출하지 않는다.
- 변경 가능 파일은 기존 write_scope 안으로 제한한다.
- 제안은 "더 일반적인 구현", "불필요한 변경 축소", "테스트가 요구한 실패 원인에 더 직접 대응"처럼 구조화된 방향으로만 제공한다.
- "새 기능 추가", "평가 기준 수정", "테스트 완화"는 금지한다.

## 후보 비교 기준

accepted 후보만 비교한다.

| 우선순위 | 비교 기준                    | 설명                                         |
| -------- | ---------------------------- | -------------------------------------------- |
| 1        | 모든 deterministic gate 통과 | 하나라도 실패하면 비교 대상 제외             |
| 2        | 원래 문제 직접성             | 수정이 objective/acceptance 원인에 직접 대응 |
| 3        | 회귀 위험 감소               | protected path 미접촉, 변경 범위 축소        |
| 4        | 테스트 품질 유지             | required test와 test-on-base 의미 보존       |
| 5        | 단순성                       | 더 작은 diff, 더 적은 파일, 더 낮은 복잡도   |
| 6        | advisory 품질 신호           | evaluator/critic 신호는 tie-breaker만 허용   |
| 7        | 비용                         | 추가 round 비용 대비 개선 근거가 명확한지    |

## 중단 조건

- refinement round 상한 도달(기본 2회)
- 새 후보가 deterministic gate 또는 hidden acceptance 실패
- 새 후보가 eval/test/hidden/protected path를 변경
- 변경 범위가 원래 문제 밖으로 확장
- advisory가 개선을 제안하지만 측정 가능한 deterministic 근거가 없음
- 기존 후보 대비 개선 근거가 미미하고 diff/위험/비용만 증가
- 같은 refinement 실패가 반복되어 human review 필요

## 잔여 리스크 / 후속 과제

- LLM/evaluator 기반 품질 판단은 advisory로만 둘 경우, "더 나음"의 일부 품질 요소는 사람이 읽는 비교 요약에 의존한다.
- refinement를 과도하게 허용하면 비용이 증가하므로 기본 round 상한과 no-benefit stop이 필수다.
- cross-loop learning과 결합하면 더 강력해지지만, 본 계획에서는 같은 문제 내부 후보 개선에만 한정한다.
