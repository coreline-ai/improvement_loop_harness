# implement_20260613_194055.md

작성 일시: `2026-06-13 19:40:55 KST`

이 문서는 이번 개발의 범위를 고정하고, 구현이 목적 밖으로 확장되지 않도록 하기 위한 작업 문서다.

## 개발 목적

현재 Skill adversarial UAT의 `context-leak` 케이스는 **UAT wrapper가 agent log를 별도 검사해서 PR 후보 branch 생성을 막는 구조**다. 그러나 VibeLoop Harness의 핵심 원칙은 `decision=accept`와 PR 후보 가능 여부가 wrapper 추론이 아니라 **커널의 deterministic gate/decision**에서 일관되게 나와야 한다는 것이다.

따라서 이번 개발의 목적은 `agent stdout/stderr 및 주요 산출물의 context/secret/hidden marker 누설`을 **core builtin guard + decision reason**으로 승격하여, 누설 후보가 wrapper 단계까지 가지 않아도 커널에서 `reject`되도록 만드는 것이다.

최종 목표:

```text
agent/artifact leak detected
  → builtin artifact/context leak gate fail
  → decision = reject
  → reason = GUARD_ARTIFACT_LEAK
  → pr_candidate = false
  → Skill/UAT/server/CLI 전 채널 동일 동작
```

## 개발 범위

- `artifact/context leak`의 고정 규칙 정의
- agent output을 raw 그대로 artifact에 저장하기 전 스캔/레드액션하는 안전 경로 도입
- `builtin:artifact-leak` 또는 `builtin:context-isolation` guard 추가
- decision engine에 전용 reason code 추가
- `eval.yaml` schema/config/type 추가
- Skill real-user/adversarial UAT의 `context-leak` 케이스를 core reject 기준으로 변경
- CLI/summary/server PR predicate가 새 reject를 자연스럽게 반영하는지 검증
- 문서와 테스트 업데이트

## 제외 범위

- LLM이 직접 context leak을 판단하는 로직 추가 금지
- hidden acceptance 테스트 원문을 Builder/Adversary/Skill에 노출하는 기능 금지
- 모든 artifact 전체를 무제한 정밀 DLP 스캔하는 대규모 보안 제품화는 제외
- 자동 merge 기능 추가 금지
- policy.lock/rulepack 전체 구현은 제외. 단, 추후 연결 가능한 config 형태로 설계한다.
- 현재 loop 중 새 규칙을 LLM 판단으로 동적 완화/강화하는 기능 제외

## 문서 기반 제약 사항

- 최종 통과 공식은 유지한다.

```text
accept = verifier_pass ∧ evaluator_pass ∧ policy_pass ∧ risk_pass ∧ provenance_pass
```

- LLM은 `accept`, `quality pass`, `selected`, `pr_candidate`를 직접 만들 수 없다.
- `ALL_PASS`는 correctness 전용 의미를 유지한다. 다만 새 guard 실패는 `ALL_PASS` 전에 reject된다.
- hidden acceptance 원문은 report/summary/PR body에 기록하지 않는다.
- agent transcript와 hidden/private context는 다른 agent 및 사용자-facing summary에 노출하지 않는다.
- UAT wrapper 판단은 최종 authority가 아니며, core kernel decision이 source of truth가 되어야 한다.

## 공통 진행 규칙

- 각 Phase는 앞선 Phase의 자체 테스트 완료 후에만 시작한다.
- 구현 중 발생한 이슈는 해당 Phase에서 수정하고 기록한다.
- 체크박스 상태를 실제 진행 상태와 맞게 업데이트한다.
- 문서에 없는 범위 확장은 하지 않는다.
- 새 guard는 deterministic rule만 사용한다. LLM 판단, advisory output, prompt self-report는 사용하지 않는다.
- raw secret/hidden marker를 새 report, gate log, PR body, summary에 재출력하지 않는다.

## 핵심 설계 결정

### 1. 이름

v1 이름은 `artifact-leak`로 한다. 이유는 `context-leak`뿐 아니라 token/secret/hidden marker 출력까지 동일 계열 문제이기 때문이다.

```yaml
- name: artifact_leak
  type: integrity
  command: builtin:artifact-leak
  required: true
```

### 2. 판정 범위

v1에서 core gate가 스캔하는 대상:

| 대상               |                                                          포함 여부 | 이유                                                          |
| ------------------ | -----------------------------------------------------------------: | ------------------------------------------------------------- |
| agent stdout       |                                                               포함 | 현재 context-leak UAT의 실제 누설 위치                        |
| agent stderr       |                                                               포함 | agent가 에러/디버그로 secret/context를 흘릴 수 있음           |
| CLI stdout JSON    |                                                          간접 포함 | agent output 기반 필드가 CLI로 전파되는지 확인                |
| eval-report        |                              직접 스캔보다는 생성 전 레드액션 보장 | report 생성 후 decision에 쓰기 어렵기 때문                    |
| gate stdout/stderr |                                   v1 제외 또는 fixed token regex만 | 프로젝트 테스트 로그까지 전체 DLP로 확장하면 오탐/비용 증가   |
| candidate patch    | 기존 diff-scope/test-integrity와 별개. v1에서는 marker scan 옵션만 | patch에 secret이 들어가는 경우 보호 필요하지만 별도 확장 가능 |

### 3. marker source

v1 marker source는 두 종류로 제한한다.

| source                                       | 예                                           | raw 값 노출 정책               |
| -------------------------------------------- | -------------------------------------------- | ------------------------------ |
| `eval.yaml artifact_leak.forbidden_literals` | 이전 issue id, UAT sentinel, 금지 context id | gate log에는 label/hash만 기록 |
| built-in sensitive regex                     | `Bearer ...`, `sk-...`, `access_token=...`   | raw match 저장 금지            |

추후 v2에서 hidden acceptance source file을 읽어 `hidden_leak_markers`를 자동 도출할 수 있지만, v1에서는 과도한 hidden 원문 처리 위험 때문에 제외한다.

### 4. raw log 저장 정책

현재 구조는 adapter가 `stdoutFile`/`stderrFile`에 raw output을 쓸 수 있다. core leak gate를 제대로 만들려면 **raw leak을 감지한 뒤에도 raw secret을 artifact에 저장하지 않아야 한다.**

v1 원칙:

```text
agent 실행 결과(raw stdout/stderr)는 메모리에서 먼저 scan
  → leak 발견 여부와 sanitized finding 생성
  → stdout/stderr artifact에는 redacted content만 저장
  → gate log/report에는 marker label, source, count, hash만 저장
```

이를 위해 `runKernel`에서 agent adapter 호출 시 `stdoutFile`/`stderrFile` 직접 전달을 제거하거나, shared `runCommand`에 redaction hook을 추가한다. 권장 경로는 전자다.

### 5. decision reason 위치

새 reason code는 guard 계열이므로 `GUARD_TEST_INTEGRITY`와 `GUARD_LIMIT_EXCEEDED` 사이 또는 직후에 배치한다.

권장 순서:

```text
6  GUARD_TEST_INTEGRITY
7  GUARD_ARTIFACT_LEAK
8  GUARD_LIMIT_EXCEEDED
9  ARTIFACT_PROVENANCE_MISMATCH
...
15 ALL_PASS
```

기존 rank 번호가 문서/테스트에 박혀 있으므로 rank를 바꾸는 대신 문서 표를 함께 갱신한다. 코드상 first-match 순서가 중요하다.

### 6. 리뷰 반영 결정 (C1·C2·C4·C5)

- **C1 — redact 항상 / reject 분리**: agent 로그의 민감 토큰/forbidden literal **redaction은 항상 수행**(안전, 손해 없음). 단 **reject 트리거는 분리**한다: `forbidden_literals` 매칭은 reject(정밀, 실제 UAT 케이스), builtin `token_like` reject는 **opt-in**(`builtins.token_like: true`)로만 동작. 기본값에서 token_like는 redact만 하고 reject하지 않는다(테스트/예제의 `Bearer`/`sk-` 오탐으로 정상 수정을 막지 않기 위함). patch 내 secret 스캔은 더 높은 심각도이나 v2.
- **C2 — bounded 캡처 위에서만 스캔**: raw 파일 스트리밍을 메모리 캡처로 바꾸되, 이미 `runCommand`의 `maxBufferBytes`(16MB)로 절단된 `result.stdout/stderr` 위에서만 스캔한다(OOM 방지). `max_scan_bytes`는 버퍼 상한 이하. timeout/agent-failure 경로의 부분 출력도 동일 redact.
- **C4 — v1은 config-gated**: `artifact_leak`가 eval.yaml에 없으면 스캔/redaction/게이트 모두 미동작(기존 동작·로그 내용 보존, 하위호환). always-on redaction은 기존 로그 소비자 영향 때문에 후속 검토.
- **C5 — leak 가드 ∈ verifier(correctness)**: `GUARD_ARTIFACT_LEAK`는 evaluator/quality가 아니라 **verifier 쪽 가드**이며 `ALL_PASS` 이전에 reject한다. §52 `accept` 합성식의 `verifier_pass`에 포함됨을 명시.

## 예상 최종 수용 기준

| 시나리오                                 | 기대 결과                                                             |
| ---------------------------------------- | --------------------------------------------------------------------- |
| 정상 agent output                        | `artifact_leak` pass, 기존 accept 가능                                |
| agent stdout에 previous issue id 포함    | `artifact_leak` fail, `decision=reject`, `reason=GUARD_ARTIFACT_LEAK` |
| agent stderr에 token-like string 포함    | `artifact_leak` fail, raw token redacted                              |
| hidden sentinel 출력                     | `artifact_leak` fail, report/summary에 raw sentinel 없음              |
| protected path + artifact leak 동시 발생 | decision 우선순위에 따라 deterministic reason 하나만 산출             |
| accepted but unqualified                 | 기존처럼 PR 후보 불가, artifact leak과 별도                           |

## 잔여 리스크 / 후속 과제

- hidden acceptance 원문 전체와 agent output의 의미적 유사도 비교는 v1 범위 밖이다. v2에서 hidden source fragment fingerprint/MinHash 등으로 검토한다.
- arbitrary regex를 허용하면 ReDoS/오탐 위험이 있으므로 v1에서는 built-in detector allowlist 중심으로 간다.
- raw log를 절대 저장하지 않는 변경은 디버깅 편의성을 일부 줄인다. 대신 redacted log와 sanitized finding을 남긴다.
- gate 로그 자체에 secret marker를 재출력하지 않는지 별도 regression test가 필요하다.
