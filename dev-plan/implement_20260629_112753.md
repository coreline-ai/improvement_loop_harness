# implement_20260629_112753.md — P1 validation tiering and full-run time control

작성 일시: `2026-06-29 11:27:53 KST`

이 문서는 P1 자연어 Skill UX 검증을 **pre-Codex routing gate**와 **live evidence lane**으로 나누어, 매 수정마다 56개 실제 Codex full run을 반복하지 않도록 고정하는 운영/개발 계획이다.

핵심 판단:

- 자연어 56개 케이스의 대부분은 먼저 "Codex 호출 전까지 올바른 intent/task/command로 라우팅되는가"를 빠르게 확인하면 된다.
- 실제 Codex가 문제를 찾고 수정하고 최종 재검증하는 경로는 이미 여러 live lane에서 반복 검증됐다.
- 따라서 평소 개발 루프는 초 단위 routing gate + 몇 분짜리 smoke/Gitea lane으로 충분하고, 56개 full-local/GitHub full은 release-grade evidence가 필요할 때만 실행한다.

## 개발 목적

- P1 자연어 Skill UX 검증 시간을 통제한다.
- 56개 자연어 케이스를 매번 실제 Codex 수정까지 실행하지 않도록 테스트 계층을 명확히 나눈다.
- Local Gitea는 PR-like 반복 검증용으로 쓰고, GitHub draft PR은 최종 evidence 용도로만 둔다.
- 문서와 테스트 명령에서 "routing PASS", "loop PASS", "Gitea PR-like PASS", "GitHub draft PR PASS"를 섞지 않는다.

## Phase 3. dedicated routing-corpus dry-run command 구현

### 목표

Phase 2에서 coverage 차이가 있으면, 실제 Codex를 호출하지 않는 56-variant corpus dry-run 명령을 추가한다.

### P0 완료 내용

P0는 실제 loop/live UAT가 실패해서 생긴 항목이 아니었다. 이미 `p1-fast`, `p1-gitea-pr`, 기존 full-local evidence는 실제 Skill orchestrator/builder 경로를 검증했다. 다만 `prompt-matrix`는 대표 routing matrix이고, `p1-coverage` 결과 live 56 corpus exact ID coverage가 32/56으로 확인됐다. 따라서 "56개 자연어 variant 전체가 실제 Codex 호출 전까지 정확히 같은 경로로 라우팅된다"는 빠른 증거가 별도로 필요했다.

구현된 dry-run은 builder, GitHub, Gitea를 호출하지 않고 기존 Skill prompt runner의 non-execute 경로만 사용한다. 즉 `run-from-prompt.mjs`가 이미 제공하는 `--execute` 없는 JSON 출력(classification, generated task/eval, command)을 56개 built-in corpus 전체에 적용한다.
