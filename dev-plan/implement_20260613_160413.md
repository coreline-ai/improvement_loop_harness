# implement_20260613_160413.md

작성 일시: `2026-06-13 16:04:13 KST`

이 문서는 이번 개발의 범위를 고정하고, 구현이 목적 밖으로 확장되지 않도록 하기 위한 작업 문서다.

## 개발 목적

VibeLoop Harness의 자율 개선 루프를 **병렬 격리 에이전트 + 고정 Verifier + 고정 Evaluator + deterministic Arbiter** 구조로 재정의하고 구현한다.

이번 계획은 직전 논의에서 드러난 중요한 오류를 바로잡는다.

- 같은 LLM이 코드를 만들고 같은 LLM이 그 코드를 "충분히 좋다"고 판단하면 자기 합리화 문제가 생긴다.
- evaluator가 advisory LLM이면 verifier를 분리해도 느슨한 판단 문제가 evaluator로 옮겨갈 뿐이다.
- 사람 승인이 hot loop에 들어가면 자율 루프가 아니라 수동 검토 루프가 된다.
- 따라서 LLM은 **탐색·생성·공격**만 하고, 통과·품질·선택은 **고정 룰**이 한다.

이 문서가 목표로 삼는 최종 루프는 다음이다.

```text
문제 1개 발견
  → task/policy/rulepack 고정
  → 독립 Builder LLM들이 병렬 후보 생성
  → 고정 Verifier가 correctness/safety 검증
  → 고정 Evaluator가 improvement-quality 평가
  → 독립 Adversary LLM들이 반례·엣지케이스·테스트 후보 공격
  → 고정 필터가 유효한 공격만 임시 gate로 편입
  → 후보 전부 재검증·재평가
  → 부족한 후보는 같은 문제 안에서 자동 refinement
  → deterministic Arbiter가 best-known candidate 선택
  → PR 후보 자동 생성
  → 다음 문제로 이동
```

중요: 이 계획은 "절대적으로 항상 세계 최고의 코드"를 수학적으로 증명한다는 뜻이 아니다. 일반 소프트웨어 변경에서 그런 증명은 불가능하다. 대신 이 프로젝트의 실현 가능한 최선 목표를 다음처럼 고정한다.

```text
best-known candidate =
  동일 task/policy/rulepack/budget/후보 풀/적대적 검증 범위 안에서
  고정 Verifier와 고정 Evaluator를 모두 통과하고,
  deterministic Arbiter 점수가 가장 높은 후보
```

## 개발 범위

- hot loop에서 사람 승인 제거: out-of-policy/high-risk는 묻지 않고 skip/backlog 처리
- `policy.lock` / `rulepack` / `task pack` 고정 계약 정의
- Builder, Adversary, Refiner 역할을 별도 exec CLI/process/worktree/artifact context로 분리
- 후보별 독립 worktree와 candidate pool 관리
- deterministic Verifier 유지 및 다중 후보 재검증 경로 추가
- deterministic Evaluator 도입: improvement-quality 고정 룰을 accept 필수조건으로 추가
- Adversary LLM의 counterexample/test proposal 생성과 고정 필터링
- proposal 통과 테스트를 ephemeral gate로 편입해 모든 후보 재검증
- refinement controller: Verifier/Evaluator/Adversary 실패 사유 기반 자동 재수정
- deterministic Arbiter: 통과 후보 중 최종 후보 선택
- PR 생성 조건을 `verified ∧ qualified ∧ selected ∧ policy_pass`로 강화
- Skill/UAT/e2e/replay fixture로 자기 합리화·느슨한 평가·같은 LLM 판단 문제 회귀 고정
- shadow rule learning: 새 룰 후보는 현재 루프에 적용하지 않고 replay/shadow 통과 후 다음 루프부터 제한적으로 자동 승격

## 제외 범위

- 자동 merge
- hot loop 중 사람 승인 요청
- LLM output만으로 accept/reject/quality pass/selection 처리
- hidden acceptance 원문을 Builder/Adversary/Skill에 노출
- candidate가 `eval.yaml`, hidden tests, rulepack, policy.lock을 완화하도록 허용
- 같은 문제 범위를 넘어선 신규 기능 자동 구현
- 절대적 "세계 최고 코드" 증명
- 모든 provider 전용 통합을 한 번에 구현하는 것
- 컨테이너/netns 완전 격리 구현은 본 계획의 필수 v1 범위가 아니라 R1 보안 워크스트림과 연결

## 문서 기반 제약 사항

- LLM은 수정 생성·반례 제안·refinement hint 생성까지만 한다.
- Verifier, Evaluator, Arbiter는 LLM이 아니라 deterministic code/rule로 구현한다.
- PR 후보 자격(상태 `accepted`)은 아래 식으로만 가능하다.

```text
accepted = verifier_pass ∧ evaluator_pass ∧ policy_pass ∧ risk_pass ∧ provenance_pass
```

- **`ALL_PASS`는 재정의하지 않는다(최종 결정).** `ALL_PASS`는 Verifier correctness 판정(= `verified`, decision rank-14 reason code)으로 그대로 둔다. Evaluator 통과는 별도 `qualified` 상태이고, 품질 실패는 decision reason code가 아니라 `quality_report.status`로 기록한다. decision engine 14규칙은 correctness 전용으로 불변. 근거·상세는 [SELF_IMPROVEMENT_LOOP_DESIGN.md](../docs/SELF_IMPROVEMENT_LOOP_DESIGN.md) §8.
- 기존 advisory critic은 남길 수 있지만 final authority가 아니다.
- 병렬 LLM 분리는 "판정자 분리"가 아니라 "탐색자/공격자 분리"다. 판정은 고정 Arbiter가 한다.
- 사람은 hot loop 심판이 아니다. 위험하거나 정책 밖인 candidate는 자동 승인 요청이 아니라 skip/backlog로 이동한다.
- 새 평가 룰은 현재 루프 중 즉시 편입하지 않는다. shadow/replay를 통과한 다음 루프부터만 적용한다.

## 공통 진행 규칙

- 각 Phase는 앞선 Phase의 자체 테스트 완료 후에만 시작한다.
- 구현 중 발생한 이슈는 해당 Phase에서 수정하고 기록한다.
- 체크박스 상태를 실제 진행 상태와 맞게 업데이트한다.
- 문서에 없는 범위 확장은 하지 않는다.
- 기존 검증 커널의 신뢰 경계를 약화하지 않는다.
- Skill, CLI, SDK, Server가 서로 다른 판정 로직을 갖지 않게 한다.
- 새 상태/이유 코드 추가 시 shared type, schema, docs, tests를 같은 Phase에서 같이 수정한다.
- 문서 변경 후 `git diff --check`를 실행한다.
- 코드 변경 Phase에서는 최소 `pnpm typecheck`, `pnpm lint`, 관련 unit/e2e 테스트를 실행한다.

| 상태           | 의미                                                | PR 후보 가능 여부 |
| -------------- | --------------------------------------------------- | ----------------- |
| `generated`    | LLM이 patch 생성                                    | 불가              |
| `verified`     | 고정 Verifier 통과                                  | 불가              |
| `qualified`    | 고정 Evaluator 통과                                 | 불가              |
| `attacked`     | Adversary proposal 필터 후 임시 gate 재검증 완료    | 불가              |
| `accepted`     | verifier/evaluator/policy/risk/provenance 모두 통과 | 후보              |
| `selected`     | Arbiter가 best-known candidate로 선택               | 가능              |
| `pr_candidate` | draft PR 생성 가능한 최종 상태                      | 가능              |
| `skipped`      | 정책 밖/위험/범위 확장                              | 불가, backlog     |

```text
verified  = builtin guards + required gates + hidden acceptance + evidence + verifier lane 통과
qualified = deterministic improvement-quality rules 통과
accepted  = verified ∧ qualified ∧ policy_pass ∧ risk_pass ∧ provenance_pass
selected  = accepted 후보 중 deterministic score/tie-break 통과
```

## Deterministic Evaluator v1 규칙

Evaluator는 LLM이 아니다. `eval.yaml` 또는 `policy.lock`에 선언된 고정 임계값과 하네스 산출물만 사용한다.

| Rule ID | 평가 기준         | 고정 판정 예시                                                                               |
| ------- | ----------------- | -------------------------------------------------------------------------------------------- |
| Q1      | evidence strength | `required_evidence` 중 task 유형에 맞는 evidence가 최소 1개 이상 `present`                   |
| Q2      | test meaning      | 새/수정 테스트는 base에서 fail, candidate에서 pass                                           |
| Q3      | target directness | changed files가 `task.metadata.target_paths` 또는 discovery fingerprint 위치와 교차          |
| Q4      | diff risk         | changed files/lines가 policy 상한 이하, forbidden/protected path 미접촉                      |
| Q5      | regression        | pass-to-pass required gates 유지, hidden acceptance 통과                                     |
| Q6      | security          | 신규 critical/high finding 0개 또는 declared threshold 이하                                  |
| Q7      | performance       | 성능 task는 baseline 대비 declared threshold 이상 개선, 일반 task는 declared regression 이내 |
| Q8      | complexity        | complexity/duplication metric이 악화되지 않거나 declared threshold 이내                      |
| Q9      | adversary closure | 필터를 통과한 critical counterexample/ephemeral gate가 모두 pass                             |
| Q10     | no gaming         | test weakening, skip, 의미 없는 assertion, eval/rule 완화, hidden test 노출 시도 없음        |

Evaluator 출력은 다음처럼 구조화한다.

```json
{
  "schema_version": "1.0",
  "candidate_id": "cand-a1",
  "status": "pass | fail | needs_refinement | inconclusive",
  "score": 87,
  "threshold": 80,
  "rules": [
    {
      "id": "Q3",
      "status": "pass",
      "metric": "target_directness",
      "value": 1,
      "threshold": 1,
      "artifact_ref": "reports/quality/directness.json"
    }
  ],
  "refinement_reasons": ["diff_too_large", "target_mismatch"],
  "rulepack_hash": "sha256:..."
}
```

## Adversary proposal 고정 필터

Adversary LLM이 만든 test/counterexample은 바로 신뢰하지 않는다. 아래 고정 필터를 모두 통과해야 임시 gate가 된다.

| 필터           | 조건                                                                           |
| -------------- | ------------------------------------------------------------------------------ |
| scope          | test target path가 허용된 테스트 디렉터리 또는 ephemeral staging 디렉터리 내부 |
| objective link | task objective, target_paths, failure fingerprint와 구조적으로 연결            |
| no weakening   | 기존 테스트 삭제/skip/느슨한 assertion/timeout 완화 없음                       |
| base/candidate | fail-to-pass 또는 pass-to-pass 기대 결과가 실제 실행으로 확인됨                |
| determinism    | 최소 2회 반복 실행에서 같은 결과                                               |
| no hidden leak | hidden acceptance 원문·경로·시크릿 문자열 포함 없음                            |
| bounded cost   | 실행 시간/파일 수/라인 수 상한 이하                                            |
| all-candidate  | 특정 후보에만 유리하게 적용하지 않고 후보 풀 전체에 동일 적용                  |

제안 분류는 다음 중 하나다.

| 분류               | 처리                                                   |
| ------------------ | ------------------------------------------------------ |
| `objective_edge`   | base fail/candidate pass이면 임시 task acceptance gate |
| `regression_guard` | base pass/candidate pass이면 임시 regression gate      |
| `quality_probe`    | metric 산출 가능하면 evaluator input                   |
| `out_of_scope`     | backlog candidate로 저장, 현재 PR에는 미편입           |
| `invalid`          | 폐기, adversary quality report에 기록                  |

## Refinement controller

Refinement는 LLM 감상이 아니라 고정 실패 사유를 만족시키기 위한 자동 재시도다.

```text
candidate fails verifier/evaluator/adversary gate
  → structured failure reasons 생성
  → 같은 objective/write_scope/acceptance/policy 안에서만 refinement task 생성
  → parent candidate patch에서 분기
  → Refiner exec CLI 실행
  → verifier/evaluator/adversary 재검증
  → accepted 후보면 candidate pool에 추가
  → round/budget/plateau 상한 도달 시 stop
```

### refinement 입력 제한

- Builder/Refiner에게 hidden test 원문을 주지 않는다.
- Adversary chain-of-thought나 전체 판단 로그를 주지 않는다.
- 구조화 실패 사유와 공개 artifact ref만 준다.
- objective와 acceptance를 다시 쓰지 않는다.
- write_scope를 넓히지 않는다.

### refinement 중단 조건

- 기본 round 상한 도달
- 모든 후보가 같은 reason으로 반복 실패
- score 개선폭이 plateau threshold 이하
- 비용/시간 budget 초과
- out-of-policy/high-risk로 분류
- valid counterexample을 해결하지 못함
- candidate가 evaluator/rule/test 완화를 시도

## Deterministic Arbiter

Arbiter는 accepted 후보만 비교한다. LLM 투표나 LLM 점수는 사용하지 않는다.

```text
candidate_score =
  evidence_score
  + quality_score
  + adversary_closure_score
  + security_delta_score
  + performance_delta_score
  + directness_score
  - diff_risk_penalty
  - complexity_penalty
  - cost_penalty
```

Tie-break 순서도 고정한다.

1. hidden acceptance와 critical ephemeral gate를 더 많이 닫은 후보
2. 더 높은 deterministic quality score
3. 더 작은 diff
4. 더 적은 파일 수
5. 더 낮은 risk area
6. 더 빠른 gate runtime
7. 후보 id 사전순 안정 정렬

Arbiter 산출물은 `reports/selection-report.json`에 저장하고 rulepack hash, candidate patch hash, evaluator report hash를 포함한다.

## Shadow rule learning 전략

자율 학습은 hot loop를 흐리면 안 된다. 따라서 새 룰은 바로 통과 기준에 섞지 않고 shadow로 돌린다.

```text
loop artifacts + failures + adversary proposals
  → rule proposal 생성
  → shadow rule로 N개 loop/replay corpus에 적용
  → known-bad를 잡고 known-good을 과도하게 깨지 않는지 확인
  → 기존 룰 완화가 아니라 append-only 강화인지 검사
  → 다음 loop rulepack candidate로 자동 승격
  → rulepack hash 기록
```

자동 승격 가능 조건:

- 기존 룰 삭제/완화 없음
- 현재 실행 중인 loop에는 적용 금지
- replay corpus에서 known-bad pass를 만들지 않음
- known-good rejection rate가 policy threshold 이하
- deterministic implementation만 포함
- rulepack diff와 hash가 artifact에 남음

불만족하면 사람에게 묻지 않고 `shadow_rejected`로 저장한다. 필요한 경우 별도 backlog issue가 된다.

## 잔여 리스크 / 후속 과제

- provider/model 독립성이 낮은 환경에서는 autonomous PR 후보 생성 신뢰도가 낮아진다. 이 경우 dry-run 또는 I2 provider 구성 가이드를 제공해야 한다.
- deterministic quality rule은 처음부터 완벽할 수 없으므로 replay corpus와 shadow rule learning이 중요하다.
- out-of-policy를 사람에게 묻지 않고 backlog로 보내면 자동 처리율은 낮아질 수 있지만, 자율성과 신뢰 경계는 유지된다.
- "최고"를 best-known으로 정의해도 hidden acceptance와 adversary coverage가 약하면 놓치는 결함이 있을 수 있다. 따라서 adversary와 replay corpus 확장이 지속 과제다.
