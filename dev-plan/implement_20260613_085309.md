## 개발 목적

VibeLoop Harness를 **Skill에 종속된 도구가 아니라 범용 AI 변경 검증 모듈**로 분리한다. 그 위에 첫 제품 채널로 `vibeloop-harness` Codex Skill을 얹어, 사용자가 자연어로 “AI가 한 문제를 수정하고, VibeLoop 기준을 통과하면 PR 후보로 정리”할 수 있게 만든다.

핵심 목표는 다음 3개다.

1. 검증 커널을 CLI 밖에서도 재사용 가능한 SDK/API 형태로 고정한다.
2. Codex OAuth proxy, task/eval template, report summarizer를 Skill과 CLI가 함께 쓰는 모듈로 분리한다.
3. Skill은 얇은 wrapper로 제품화하여, 이후 CLI/CI/GitHub Action/API/PR Bot/자율 루프가 같은 core를 재사용할 수 있게 한다.

## 문서 기반 제약 사항

- 최종 판정은 LLM이 아니라 deterministic decision engine의 `ALL_PASS`다.
- Skill은 core 로직을 재구현하지 않고 CLI/SDK를 호출하는 thin wrapper여야 한다.
- `task.yaml`, `eval.yaml`, `eval-report.json`은 외부 제품 채널과 연결되는 contract boundary다.
- OAuth token, `auth.json`, API key 원문은 Skill/script/report/log에 저장하지 않는다.
- hidden acceptance는 agent/Skill에 노출하지 않는다.
- PR 생성은 `accept/ALL_PASS` 또는 human-approved 후보만 가능하다.

- Skill 제품화 후 marketplace/배포 형식은 별도 packaging 계획이 필요하다.
- `@vibeloop/sdk`가 안정화되기 전 외부 API로 공개하면 breaking change 위험이 있다.
- 다양한 언어 template은 초기 Node/Python/Web 이후 실제 repo 사례를 보며 확장해야 한다.
- 완전 자율 discovery 루프는 이번 범위가 아니라 SDK 안정화 후 `AUTONOMOUS_LOOP_SPEC.md` 기준으로 별도 계획을 세운다. 이번 Phase 8은 deterministic issue queue로 Skill 실사용 경로를 검증하는 UAT다.
