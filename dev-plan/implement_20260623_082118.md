이 문서는 작업 범위를 다시 고정한다. 현재 목표는 CI artifact, broad corpus, semantic/M4 corpus를 계속 넓히는 것이 아니라, **내부 개선 루프가 실제 코드 문제 하나를 선택하고, 후보를 만들고, 결정론 검증으로 최종 개선분을 고른 뒤, 그 선택된 개선분만 PR 출판 도구로 내보내는 흐름**을 확인하고 강화하는 것이다.

## 개발 목적

- VibeLoop의 목적은 GitHub나 CI가 아니라 **내부 코드 개선 루프**다.
- 루프는 문제를 한 번에 하나씩 다룬다.
- 각 문제는 후보 생성, 검증, 선택, 최종 재검증을 거쳐야 한다.
- PR은 내부 루프가 고른 최종 개선분을 사람이 검토하기 쉽게 내보내는 도구일 뿐이다.
- 다음 작업은 "증거 체계 확장"이 아니라 "실제 루프 실행으로 내부 코드 개선을 하나씩 완성"하는 것이다.

## 개발 범위

- 현재 repo 내부에서 실제 개선 대상 issue 1개를 확정한다.
- 해당 issue의 task/eval/검증 조건을 명시한다.
- `improve` 또는 필요한 내부 루프 드라이버가 후보 생성, 검증, 선택까지 도는지 확인한다.
- 선택된 최종 개선분만 branch로 분리한다.
- GitHub draft PR은 선택된 개선분 출판 단계에서만 사용한다.
- PR diff와 local evidence가 동일한 최종 선택 patch를 가리키는지 확인한다.
- 다음 issue는 현재 issue의 evidence가 닫힌 뒤에만 진행한다.

## 제외 범위

- CI artifact 재현성 개선을 새 기능으로 진행하지 않는다.
- broad real project corpus를 더 확장하지 않는다.
- semantic/M4 corpus를 새 domain으로 더 확장하지 않는다.
- Product-100 전체 corpus를 기본 작업 목표로 삼지 않는다.
- GitHub draft PR 생성 자체를 제품 목적처럼 다루지 않는다.
- 여러 문제를 한 번에 묶어 처리하지 않는다.
- hidden/adversary 원문을 agent나 PR에 노출하지 않는다.

## 공통 진행 규칙

- 한 번에 issue 1개만 다룬다.
- issue 선택 전에는 구현하지 않는다.
- 후보 생성/검증/선택 evidence 없이 PR을 만들지 않는다.
- PR은 transport/publish 단계일 뿐이며 accept 근거가 아니다.
- CI는 회귀 확인 도구일 뿐이며 작업 목적이 아니다.
- corpus 확장은 사용자가 다시 명시하기 전까지 진행하지 않는다.
- 각 phase는 자체 테스트와 완료 조건을 가진다.
