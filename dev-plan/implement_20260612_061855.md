## 개발 목적

- LLM은 **수정자/제안자** 역할로 제한하고, 최종 통과 판정은 deterministic harness가 수행하도록 신뢰 경계를 강화한다.
- `eval-report.json`의 `decision: accept`가 단순한 "LLM이 괜찮다고 판단함"이 아니라, 독립 검증·증거·재현 가능한 게이트를 통과했다는 의미가 되도록 만든다.
- prompt injection, self-preference bias, overreliance, test gaming, eval 파일 변조, PR 스팸을 명시적으로 방어한다.
- 자동 루프의 산출물은 **auto-merge가 아니라 draft PR**이며, 위험 변경은 human review로 라우팅한다.

## 개발 범위

- 통과 기준 신뢰 경계 강화
  - LLM critic/adversarial review는 advisory로만 유지 (현행 유지 + 회귀 고정)
  - final decision은 deterministic decision engine만 수행 (현행 유지 + 회귀 고정)
  - `eval-report.json`에 **hash 기반 provenance** 기록·검증 추가 (서명은 범위 외 — 후속 과제)
- 증거 기반 검증 강화
  - 기존 test-on-base 메커니즘을 `fail_to_pass`/`pass_to_pass` 그룹으로 **형식화** (신규 구현 아님)
  - `hidden_acceptance`(held-out) 테스트 그룹 **신규** 도입 — 보관·주입·redaction 운용 규칙 포함
- 독립 verifier lane 추가
  - 061653이 만든 CI 위에 verifier 의미론(비교 대상·mismatch 처리)을 정의
  - "독립"은 **환경 독립**이다(같은 deterministic engine을 다른 환경에서 실행 — 로컬 환경 조작·결과 변조 탐지가 목적). 판정 로직 독립으로 과대 표기하지 않는다.
- prompt injection 방어의 **gap 보강**
  - 구조화 추출·원문 비전달은 구현 완료 — source별 trust level, injection indicator 라우팅만 신규
- 리포트/웹/API 표시 강화
  - deterministic / advisory / provenance / verifier 구분 표기, human review 사유 reason code 고정

## 제외 범위

- 자동 merge 구현은 제외한다. 모든 자동 산출물은 draft PR까지만 허용한다.
- LLM judge를 최종 합격 판정자로 승격하지 않는다.
- target repository의 테스트 파일을 신뢰 기준 자체로 삼지 않는다. 테스트 변경은 별도 integrity 검증 대상이다.
- **eval-report 디지털 서명은 제외한다** — 단일 머신 자기서명은 변조 방어 가치가 낮다. hash provenance까지만 하고, 서명은 외부 verifier(별도 신뢰 도메인) 도입 시 후속 과제.
- **candidate의 repro command "실행"은 제외한다** — untrusted 입력에서 추출한 명령 실행은 injection 벡터다. 표시 전용 필드까지만 (Phase 5).
- 모든 외부 SaaS verifier 연동을 즉시 구현하지 않는다. MVP는 로컬/CI 독립 검증 계약과 확장 포인트를 우선 만든다.
- 대규모 agent framework 교체나 모델 벤더 종속 기능 추가는 제외한다.

## 고정 통과 의미

| 항목 | 통과 의미 | 최종 authority |
|---|---|---|
| 수정 생성 | Agent/LLM이 후보 patch 생성 | 아님 |
| 비판/적대적 리뷰 | LLM critic 또는 static reviewer가 문제 제기 | 아님, advisory |
| 테스트 실행 | 지정 gate command 결과 수집 | 부분 authority |
| 증거 검증 | 신규 테스트·보안 개선·성능 개선 evidence 탐지 | deterministic detector |
| 보호 경로 검증 | eval/schema/guard/protected file 변조 차단 | builtin guard |
| 최종 판정 | `decision: accept`, `reason: ALL_PASS` 산출 | deterministic decision engine |
| PR 생성 | accepted 또는 human approved 후보만 draft PR 생성 | orchestrator policy |

최종 `accept` 조건은 아래 조합을 모두 만족해야 한다.

```text
builtin guards pass
+ required gates pass (hidden_acceptance 포함 — required gate로 모델링)
+ required evidence present
+ fail-to-pass evidence 검증
+ pass-to-pass regression 유지 (required gate + test integrity로 보존)
+ protected path/write_scope 위반 없음
+ risk low/known 또는 human approval
+ eval-report schema/provenance(hash) 검증
= decision: accept / ALL_PASS
```

이 표는 Phase 1에서 [EVAL_ENGINE_SPEC.md](../docs/EVAL_ENGINE_SPEC.md)로 승격해 명세의 일부로 만든다.

## 잔여 리스크 / 후속 과제

- hidden test는 운영상 보안 저장소/CI secret과 연결되어야 하므로, 로컬 개발 문서만으로 완전 해결되지 않는다.
- eval-report **디지털 서명**은 외부 verifier(별도 신뢰 도메인) 도입 시 후속 과제 — 단일 머신 자기서명은 가치가 낮아 본 계획에서 제외.
- `repro_command` 실행 지원은 명령 템플릿 화이트리스트 설계가 선행되어야 하는 별도 워크스트림.
- target repo 자체가 악성 스크립트를 포함할 수 있으므로 장기적으로 container/network sandbox가 필요하다.
- LLM judge를 완전히 배제해도 테스트 oracle 자체가 부실하면 accept 의미가 약해질 수 있다.
- strict verifier 모드는 비용과 속도를 증가시키므로 프로젝트별 policy preset이 필요하다.
- CI verifier 결과의 자동 수집(webhook)은 수동 첨부 계약이 검증된 후 후속.
